## 1. Архітектура та інфраструктура

Один майстер-садівник на Ельбі, у полі, зі слабким або відсутнім звʼязком, інколи з помічником. Це **не CRM, а робочий журнал ремісника**: ранок планую — день працюю — ввечері «закриваю каси». Уся архітектура підпорядкована одному принципу: **простий фасад, глибока інфраструктура**. Нижче — РІШЕННЯ, не варіанти.

> **Канон і застарілі специфікації.** Єдине канонічне джерело істини продукту — це
> цих **два документи** (`canone/01-visione-esperienza-utente.md` + `canone/02-specifica-canonica.md`).
> Старі файли `docs/CANONE.md` і `docs/PRD-CANONE.md` — це **видобуток домену старого
> проєкту** (звідки перенесено лише робочий контекст: терміни, режими, рушій грошей).
> Вони **офіційно застарілі** й **не є каноном**: де вони суперечать цим документам —
> праві ці. Зокрема їхній словник несумісний (`Collaboratore` як корінь сутності замість
> `Operatore`; відсутні `giornate`/`totale`; «Центр» замість «Soldi»). Цей документ
> приймає **новий словник** (`Operatore`/`«io»`, `svolto`/`programmato`, чотири `modalità`,
> `Soldi`), і всі відповідності зведено в глосарії §10.3.

### 1.2 Рішення про бекенд / БД

**РІШЕННЯ: Local-first PWA на одному локальному джерелі істини, із заздалегідь закладеним синком — стартувати без сервера.**

Чому local-first, а не одразу керований бекенд (Supabase/Postgres):

| Аргумент | Суть |
|---|---|
| Профіль користувача | Один майстер у полі зі слабким звʼязком. Local-first = **миттєвість і офлайн за замовчуванням**; план уранку та «закриття кас» працюють у мертвій зоні. Сервер тут — постійна точка відмови саме там, де працює користувач. |
| Ментальна модель | «Усі мої дані — мій зошит у телефоні» простіше за «дані на сервері, потрібен інтернет і логін». Нуль схем, рядків підключення, RLS на старті. |
| Один користувач | Класична проблема local-first (мультиплеєр-мерджі) тут вироджена. Помічник — рідко й асинхронно → покривається простим синком, не CRDT з першого дня. |
| Вартість/експлуатація | Нуль серверів, нуль рахунків, нуль бекапів на нас (експорт-файл = бекап). |

**Конкретна форма:**

| Аспект | Рішення |
|---|---|
| Сховище | **IndexedDB через Dexie.js** — структуровані таблиці, індекси, обсяг, транзакції. `localStorage` — лише дрібні налаштування. |
| Доменний шар | **Деривати рахуються, не дублюються** (`codice parlante`, тотали, статус оплат у `src/lib/`). БД зберігає лише **факти-події**, не агрегати. |
| Стан застосунку | **Zustand** як UI-стор; джерело істини — Dexie, Zustand гідрується з нього. |
| Експорт/імпорт | Один файл (JSON) — водночас бекап і ручний перенос між пристроями. |

### 1.3 Офлайн / персистенс

- **Офлайн-перший за замовчуванням:** усі читання й записи йдуть у локальний Dexie/IndexedDB; мережа не потрібна для жодної основної дії.
- **PWA-кеш (Workbox):** оболонка застосунку, шрифти й ассети кешуються — повторний запуск працює без звʼязку.
- **Бекап:** експорт-файл JSON — ручний бекап і перенос на старті; автоматичних хмарних бекапів у v1 **немає** (поза v1).
- **Закладки під синк уже на старті** (дешеві, окупаються):

| Закладка | Навіщо |
|---|---|
| `id` як **UUID/ULID**, генеровані на клієнті | Дані зливаються без перенумерації при майбутньому синку. |
| `updatedAt` + `deleted`-tombstone + версія/монотонний лічильник | Фундамент синку та «м'яких видалень». |
| Модель = **події/факти**, агрегати деривуються | Спрощує і конфлікти, і майбутню реплікацію. |

### 1.4 Шар даних із заміною на HTTP без зміни UI

Архітектурний шов закладається **сьогодні**: UI спілкується лише з діями стора, що ходять через **інтерфейс репозиторію**; реалізація репозиторію змінна.

```
+---------+     +-------------------+     +--------------------------+     +------------------+
|   UI    | --> | Дії стора         | --> | Інтерфейс репозиторію    | --> | Реалізація       |
| (React) |     | (Zustand actions) |     | (контракт, незмінний)    |     | v1: Dexie        |
+---------+     +-------------------+     +--------------------------+     | v2: Dexie + синк |
                                                                          | v3: HTTP-клієнт  |
                                                                          +------------------+
```

Принцип AGENTS.md дослівно: **щоб під'єднати бекенд, досить замінити реалізацію дій — інтерфейс лишається ідентичним.** Жоден екран, компонент чи дія стора не переписується; змінюється лише тіло методів репозиторію.

**Етапи без переписування UI:**

1. **v1 (зараз):** Dexie локально + експорт-файл як бекап. Нуль інфраструктури.
2. **Синк-шар** (коли треба мультидевайс/помічник): синхронізаційний рушій **поверх того ж інтерфейсу репозиторію** —
   - легкий керований local-first синк (**PowerSync / ElectricSQL / Replicache-подібний**), що синкає IndexedDB ↔ Postgres у фоні; сервер стає лише «дзеркалом»;
   - або **Supabase/Postgres** як бекенд, до якого ходить репозиторій, зберігаючи локальний кеш як офлайн-буфер.
3. **Повний бекенд** (якщо колись виросте в команду): та сама заміна реалізації репозиторію на HTTP-клієнт.

### 1.5 Нефункціональні вимоги

| Вимога | Цільовий стан у v1 |
|---|---|
| **Швидкість** | Усі основні дії — миттєві (локальне читання/запис, без мережевого циклу). Без важких растрів-текстур; grain через CSS/SVG. |
| **Офлайн** | Повна працездатність без звʼязку: план, робота, «закриття кас», перегляд історії — усе локально. |
| **PWA** | Інстальований застосумок (Vite PWA / Workbox), офлайн-кеш оболонки, splash з обкладинкою. |
| **Мобіл-фьорст** | Великі тач-цілі (рукавиці/бруд), нижня досяжна зона для головних дій. |
| **Доступність** | Radix-примітиви; `reduced-motion` має статичний дублікат кожного руху. |
| **Поза v1 (явно)** | Хмарний синк і мультидевайс, акаунти/авторизація на сервері, автоматичні хмарні бекапи, CRDT-мерджі, командний доступ. |

**Підсумок:** старт — **нуль серверів, офлайн-перший, файл-бекап**; цільова глибина — **прозорий синк IndexedDB ↔ Postgres** за тим самим інтерфейсом репозиторію. Фасад лишається простим («мій зошит»), інфраструктура поглиблюється непомітно для користувача й без переписування UI.


## 2. Доменна модель

Доменний шар «Albero Maestri» — це чистий TypeScript у `src/lib/`, без backend. Усі дати — ISO-рядки `yyyy-mm-dd`; грошова арифметика всюди проходить через `arrotonda(n)` = `Math.round((n + Number.EPSILON) * 100) / 100`; епсілон-поріг для порівнянь грошей/залишків — `0.005`. Дані живуть у `localStorage` під коренем `Database`.

### 2.1. Принцип каноном: derived-not-stored

РІШЕННЯ: похідні дані (`codice parlante`, всі тотали, стани оплат, історія) **не зберігаються** — вони перераховуються при читанні з функцій у `src/lib/`. Сутності зберігають лише первинні факти + «снапшоти» (заморожені тарифи), решта — формули. Це наскрізна конвенція всієї моделі.

### 2.2. Перелік сутностей і полів (`types.ts`)

Легенда типів: `string` за замовчуванням; `?` — опціональне поле; `€` — `number` (гроші); `ISO` — дата-рядок `yyyy-mm-dd`.

### 2.3. Codice parlante — деривований ідентифікатор клієнта

`codice` (`codice-parlante.ts`) **не є полем** — зберігаються лише `inizialiCodice`. Формат `II-GG-SS-AA` (напр. `MR-03-12-05`):

| Частина | Сенс | Джерело |
|---|---|---|
| `II` | ініціали (велика літера імені або `X` + перша літера прізвища) | `inizialiCodice` (фіксована) |
| `GG` | середня к-ть днів на інкасацію | з `pagati` платежів |
| `SS` | середня витрата / `UNITA_SPESA` (`50` €) | з `pagati` платежів |
| `AA` | повних років співпраці | з найранішої роботи |

`GG`/`SS`/`AA` завжди обчислюються при читанні. Колізії ініціалів вирішуються лічильником: база `MR` → зайнято → `MR1`, `MR2`…

Це навмисне спрощення — `misto`/`in_corso` лишаються лише для відображення legacy-даних.

## 3. Розрахунки, рушій грошей і машини станів

Цей розділ фіксує канонічний рушій грошей: як з полів `Lavoro` дериваційно
обчислюються всі економічні величини, як працюють машини станів `fase` та
`statoIncasso`, і як формується та присвоюється «codice parlante». Усе тут —
**деривація на читання**: жодне агреговане значення не зберігається.

### 3.0 Базові правила арифметики

| Правило | Значення | Призначення |
|---|---|---|
| `arrotonda(n)` | `Math.round((n + Number.EPSILON) * 100) / 100` | округлення кожної грошової величини до 2 знаків |
| Епсілон-поріг | `0.005` | усі порівняння грошей/залишків («майже нуль», «майже рівні») |
| Дати | ISO `yyyy-mm-dd` | єдиний формат збереження |
| Доба в мс | `86_400_000` | різниці дат |

**РІШЕННЯ:** грошова арифметика **завжди** проходить через `arrotonda`;
порівняння — через поріг `0.005`, ніколи через `===` на числах з плаваючою
комою.

### 3.1 Три ортогональні осі та «4 режими»

Робота описується трьома незалежними осями. Кожна має канон-поле і
legacy-fallback (зворотна сумісність із seed):

| Вісь | Канон-поле | Значення | Fallback, якщо поле відсутнє |
|---|---|---|---|
| Часова | `l.fase` | `"fatto"` \| `"da_fare"` | `l.stato === "fatto" ? "fatto" : "da_fare"` |
| Компенсація | `l.modo` | `"preventivo"` \| `"ore"` | `l.tipoCompenso === "ore" ? "ore" : "preventivo"` |
| Підрахунок годин | `l.conteggio` | `"totale"` \| `"per_giorni"` | `"totale"` |

«4 режими» з ТЗ — це дві значущі осі розрахунку:

```
            ВІСЬ КОМПЕНСАЦІЇ (modo)         ВІСЬ ГОДИН (conteggio)
          +-----------------------+      +--------------------------+
          | preventivo: lordo =   |      | totale:     години на    |
          |   l.prezzo            |      |   учасника одним числом  |
          +-----------------------+      +--------------------------+
          | ore: lordo =          |      | per_giorni (giornate):   |
          |   oreCliente *        |      |   години розписані по    |
          |   tariffaCli         |      |   днях (RigaGiorno.ore)  |
          +-----------------------+      +--------------------------+
```

| «Режим» (термін ТЗ) | Вісь | Дія в рушії |
|---|---|---|
| `preventivo` | `modo` | `lordo = arrotonda(l.prezzo ?? 0)` |
| `ore` | `modo` | `lordo = arrotonda(oreCliente * tariffaCli)` |
| `totale` | `conteggio` | години на учасника як одне число (`PartecipanteLavoro.oreTotale`) |
| `giornate` (`per_giorni`) | `conteggio` | години розписані по днях (`RigaGiorno.ore`: `collaboratoreId -> години`) |

**КАНОН (джерело істини для годин):** попри поля `oreTotale`/`RigaGiorno`,
фактичні години для розрахунку **завжди** беруться з `db.ore`
(`RegistrazioneOre`), де `lavoroId` і `operatoreId` збігаються. `conteggio` й
`partecipanti` несуть лише снапшот тарифу-собівартості і список тих, хто
працював; `oreTotale`/`RigaGiorno` — це поля вводу/UI, а не розрахунку.

### 3.2 `calcoloLavoro(db, l)` — єдине джерело економіки роботи

Вхідні: `ioId = operatoreIo(db)?.id`; `fase`, `modo` (з §3.1);
`base = partecipantiCalc(db, l)` (список учасників з `ore` та `tariffa`, де
`tariffa = p.tariffaSnapshot ?? тариф_оператора`; legacy-гілка — завжди
актуальний тариф).

**ТОЧНІ ФОРМУЛИ (порядок обчислення):**

```
oreTotali         = arrotonda( Σ base.ore )
oreCliente        = oreTotali
tariffaCli        = l.tariffaClienteSnapshot
                    ?? db.clienti[clienteId].tariffaOraria ?? 0

lordo             = modo === "preventivo"
                      ? arrotonda( l.prezzo ?? 0 )
                      : arrotonda( oreCliente * tariffaCli )

для кожного p ∈ base:
    escludi       = (p.collaboratoreId === ioId) && !l.contaMieOreComeCosto
    p.costo       = escludi ? 0 : arrotonda( p.ore * p.tariffa )

costoCollaboratori = arrotonda( Σ p.costo )
speseTotali        = arrotonda( Σ db.spese[lavoroId = l.id].importo )

incassato          = fase === "da_fare"
                       ? 0
                       : arrotonda( Σ db.pagamenti[lavoroId = l.id].importoIncassato )

netto              = arrotonda( lordo − speseTotali − costoCollaboratori )
daIncassare        = arrotonda( max(0, lordo − incassato) )
```

| Величина | Сенс |
|---|---|
| `lordo` | брутто / виручка з роботи (виставлено клієнту) |
| `costoCollaboratori` | собівартість праці (учасники, опц. + «io») |
| `speseTotali` | прямі витрати, прив'язані до роботи |
| `incassato` | реально отримано (0, якщо робота ще `da_fare`) |
| `netto` | прибуток = `lordo − spese − costoCollaboratori` |
| `daIncassare` | залишок до інкасації = `max(0, lordo − incassato)` |

### 3.3 Правило `contaMieOreComeCosto`

Default — `false`. Керує лише тим, чи входять години оператора «io» в
`costoCollaboratori`:

```
                      "io"-години         інші учасники
                  +-------------------+ +-------------------+
  у lordo?        |       ТАК          | |       ТАК          |
                  +-------------------+ +-------------------+
  у costo?        | false → НІ (=0)    | |  ТАК завжди        |
                  | true  → ТАК        | |  ore * tariffa     |
                  +-------------------+ +-------------------+
```

- **`false` (за замовчуванням):** мої години → `costo = 0`. Мій час входить у
  `lordo`, але не зменшує `netto` → це чистий прибуток.
- **`true`:** мої години оцінюються за `tariffaSnapshot`/тарифом і додаються до
  `costoCollaboratori`, зменшуючи `netto`.
- Виключення застосовується **лише** до учасника з `collaboratoreId === ioId`.
  Усі інші співробітники завжди дають `costo = arrotonda(ore * tariffa)`.

`operatoreIo(db)` = `db.operatori.find(o => o.ruolo === "titolare")` (fallback —
перший оператор). За каноном присутній завжди; у seed його `tariffaOraria` =
`null` (= `0` у розрахунках).

### 3.4 Машина станів `fase`

Часова вісь має два стани й одну канонічну конверсію:

```
        ┌──────────────┐   «фіксація виконання»   ┌────────────┐
        │  programmato │ ───────────────────────▶ │   svolto   │
        │  (da_fare)   │ ◀─────────────────────── │  (fatto)   │
        └──────────────┘   «повернути в план»     └────────────┘
```

| Стан `fase` | Legacy `stato` | Сенс | Вплив на рушій |
|---|---|---|---|
| `da_fare` (`programmato`) | `da_fare` / `in_corso` | заплановано | `incassato = 0` примусово |
| `fatto` (`svolto`) | `fatto` | виконано | `incassato` рахується з `db.pagamenti` |

**КАНОН конверсії:** доки `fase === "da_fare"`, `calcoloLavoro.incassato` = `0`
**незалежно** від наявних платежів. Інкасація має сенс лише на роботі у фазі
`fatto`. Конверсія `da_fare → fatto` синхронізує legacy-поле `stato` (вони
тримаються узгодженими).

**Поза v1:** проміжний legacy-стан `in_corso` у канон-осі не відтворюється —
дериваційно він зливається у `da_fare`.

### 3.5 Машина станів `statoIncasso` (per-lavoro)

Деривується в `calcoloLavoro` з `incassato` vs `lordo`, поріг `0.005`:

```
  incassato ≤ 0.005 ───────────────────────────▶  non_pagato
  incassato ≥ lordo − 0.005 ───────────────────▶  pagato
  інакше ──────────────────────────────────────▶  parziale
```

| `StatoIncasso` | Умова |
|---|---|
| `non_pagato` | `incassato <= 0.005` |
| `pagato` | `incassato >= lordo − 0.005` |
| `parziale` | проміжне |

**Каскад «Incassa subito»:** позначити відкритий платіж зінкасованим
(`importoIncassato = importoAtteso`, `dataIncasso = oggi`) → `statoIncasso`
переходить у `pagato`, платіж одразу з'являється в русі коштів (incasso,
`segno +1`). Відкритий платіж знаходить `pagamentoApertoLavoro(db, lavoroId)`
(перший де `importoAtteso − importoIncassato > 0.005`).

**Окрема, не плутати** — стан платежу-як-інвойсу `StatoPagamento`
(`conti.statoCalcolato`): `in_attesa` / `pagato` / `in_ritardo`, залежить від
`importoAtteso`, `importoIncassato`, `dataScadenza`, `oggi`. `statoIncasso` —
шкала per-lavoro; `StatoPagamento` — шкала per-pagamento.

### 3.6 «Codice parlante»: формат і присвоєння

**Формат:** `II-GG-SS-AA` (приклад `MR-03-12-05`).

```
   M R - 0 3 - 1 2 - 0 5
   │ │   │ │   │ │   │ │
   └─┴── ініціали (фіксовані, inizialiCodice)
         └─┴── GG: середні дні до інкасації
               └─┴── SS: середня витрата/робота ÷ UNITA_SPESA (50 €)
                     └─┴── AA: повних років співпраці
```

`GG`/`SS`/`AA` **ЗАВЖДИ обчислюються з даних, ніколи не зберігаються**. Кожна
частина → `due(n)` = `Math.round`, **clamp `[0..99]`**, `padStart(2,"0")`.

**Крайові випадки сегментів (канон, 2 цифри — свідома межа дизайну):**

| Випадок | Поведінка |
|---|---|
| Сегмент `> 99` (борг >99 днів, лаворо >9 900 €, >99 років) | clamp до `99` — показуємо «`99`», не переповнюємо формат |
| `GG` за передоплати (`dataIncasso < dataEmissione`) | `giorni = max(0, …)` → `00`, **ніколи не відʼємне** |
| Немає даних (нуль `pagati`/робіт) | відповідний сегмент = `00` — це чесний стан, не помилка |

Формат фіксовано на 2 цифри/сегмент: це усвідомлена межа дизайну (жетон лишається
коротким і читабельним), а не недогляд.

**Присвоєння ініціалів (`II`):**
- `inizialiDa(nome, cognome)` = перша літера імені (велика, або `X`) + перша
  літера прізвища.
- `assegnaIniziali(...)` — лічильник колізій: база `MR`; якщо зайнято → `MR1`,
  `MR2`… (перший вільний серед наявних `inizialiCodice` з тією базою). Фіксується
  один раз у `cliente.inizialiCodice`.

**Обчислення частин (`calcolaCodice`, по платежах `pagato`):**

| Частина | Формула |
|---|---|
| `GG` | по pagati з `dataIncasso`: `giorni = max(0, (dataIncasso − dataEmissione)/86_400_000)`; `gg = Σgiorni / N`. Немає → `0` |
| `SS` | по pagati: `totale = Σ importoIncassato`; `nLavori = max(1, унікальні lavoroId)`; `ss = totale / nLavori / 50`. Немає → `0` |
| `AA` | є `dataPrimoLavoro` → `floor((oggi − dataPrimoLavoro) / (365.25 * 86_400_000))`; інакше `0` |

`codiceCliente(db, id)` бере платежі клієнта зі `stato === "pagato"`,
`dataPrimoLavoro` = найраніша `data` його робіт; немає клієнта →
`"??-00-00-00"`. `leggiCodice(codice)` — зворотне розкладання (`spesaMedia = SS *
50`).


## 4. Навігація, IA, прогресивне розкриття

Розділ фіксує канонічну інформаційну архітектуру (IA) «Albero Maestri»: карту
екранів, нижню навігацію, просторову модель, патерни прогресивного розкриття,
перетікання між режимами та єдину глобальну дію створення запису. Усе нижче —
**рішення**, а не варіанти. Те, що поза v1, позначено явно.

### 4.0. Дві осі, на яких тримається IA

Уся структура виводиться з двох ортогональних осей:

| Вісь | Полюси | Природа |
|---|---|---|
| Вісь **типу запису** | `svolto` (зроблено / приймаю гроші) ↔ `programmato` (план) | горизонтальна, два світи перетікають |
| Вісь **простору застосунку** | `Dashboard` (ліворуч) ↔ `Agenda` (центр-дім) ↔ `Anagrafiche` (праворуч) | просторова мапа, керується свайпами |

`Soldi` стоїть **осторонь обох осей** — це не «місце на мапі», а наскрізний зріз
грошей. Тому `Soldi` живе в нижній навігації, а `Dashboard` — ні (див. 4.2).

Доменна вертикаль (канонічний словник):
`cliente` → `operaio` (за замовчуванням = власник `«io»`) → `lavoro`.
Усе починається зі створення `lavoro` через глобальну дію `«+»`.

### 4.1. Карта екранів

```
                    ┌───────────────────────────────────┐
```

### 4.2. Нижня навігація — рішення

**Таб-бар = 3 блоки + центральний FAB.** Канонічний склад:
`Agenda` (дім) · `Soldi` · `Anagrafiche`, а `«+»` — посередині над ними.

```
   ┌──────────────┬───────────────────────┬──────────────┐
   │   AGENDA     │     ( + )  FAB-дія     │  ANAGRAFICHE │
   │   дім        │   «Crea registrazione» │              │
   ├──────────────┴───────────────────────┴──────────────┤
   │              SOLDI   (наскрізний зріз)               │
   └─────────────────────────────────────────────────────┘
```

Рішення та обґрунтування:

- **`Dashboard` свідомо ПОЗА навігацією.** Це не «куди йду», а «що видно
  зараз» — аналітичний провал. Два природні входи: (1) свайп ліворуч з `Agenda`;
  (2) tap по будь-якому підсумковому блоці в `Soldi` (`guadagnato`/`incassato`/
  `da incassare`). Четвертий таб був би сміттєвим дублем → не існує.
- **`Soldi` Є в навігації, хоча `Dashboard` — ні.** `Soldi` — це **дія**
  (інкасую / плачу), до неї має бути прямий шлях звідусіль. `Dashboard` —
  споглядальний, йому достатньо контекстних входів.
- **`Agenda` — дім, а не рядовий пункт.** Користувач не «перемикається на
  Agenda», він *повертається* до неї; tap по будь-якій іконці й назад = `Agenda`.
- **Усе клікабельне, нуль кнопок «детальніше».** Картка → `lavoro`; чіп клієнта →
  `scheda cliente`; чіп операйо → `scheda operaio`; блок `Soldi` → `Dashboard`.

### 4.3. Просторова модель та свайпи

- **Горизонтальний свайп** на `Agenda` ковзає по просторовій мапі
  `Dashboard <-> Agenda <-> Anagrafiche`. Таб-бар перемикає «режими застосунку»
  (`Agenda`/`Soldi`/`Anagrafiche`) — два механізми не конфліктують.
- **Peek-край:** сусідній екран визирає на 12-16px зліва/справа від `Agenda` —
  навчає жесту без підказок.
- **Бейдж-бульбашка дати** в `Agenda` у стилі Telegram: при скролі плаває й
  оновлюється на дату, що наближається; записи розбиті по даті + назві дня.
- **`Soldi`: внутрішній toggle** `da incassare` / `da pagare` — той самий
  segment-slide, що і всюди (див. 4.5), без свайпу між екранами.

### 4.4. Патерни прогресивного розкриття

Закон: **«порожнє поле не існує»** — поле з'являється тільки коли його зміст став
актуальним; глибша деталь живе на власному екрані.

```
ЕКРАН «Crea registrazione» — компактний кістяк, що росте знизу:

┌─────────────────────────────────────────────┐
│  [ svolto ]  [ programmato ]   ← розвилка №1  │
│                                              │
│  Cliente:  ( + scegli / nuovo )   ← рядок-   │
│  Operaio:  «io»  ▾                  тригер    │
│                                              │
│  Modalità di calcolo:                        │
│  ( preventivo · ore · giornate · totale )    │
│            ▲ вибір розкриває різні поля:      │
└─────────────────────────────────────────────┘
   preventivo → [ importo ] + [ periodo: dal—al ]
   ore        → [ quante ore ]
   giornate   → [ + aggiungi riga: data · ore ] (рядки, Σ авто)
   totale     → [ quantità totale ]  (без розбивки по днях)
```

| # | Патерн | Канонічне правило |
|---|---|---|
| А | **Розвилка типу = вибір кімнати** | `svolto` vs `programmato` — перше й головне рішення на `Crea`; поки не обрано — нічого зайвого |
| Б | **Модаліте як стрілка** | `preventivo`/`ore`/`giornate`/`totale` диктують поля; поля інших режимів фізично відсутні до вибору |
| В | **Тригер-рядок замість поля** | `Cliente`, `spese`, «ho lavorato con qualcuno» — спершу один рядок-натяк; не вписав витрат → секції `spese` немає |
| Г | **`operaio` за замовч. схований** | стоїть `«io»`; поля `[mie ore | sue ore]` з'являються лише при додаванні другого `operaio` |
| Д | **Спливна модалка для рідкісного** | `lavoro` без клієнта → при збереженні модалка `«imposta tariffa»`; вибір наявного клієнта → окремий екран-список → повернення зі збереженим станом форми |
| Е | **Деталь — завжди власний екран** | картка `Agenda` показує лише `ore svolte · incassato · da incassare · cliente + chip operai · azioni`; решта — у `Dettaglio lavoro` |
| Є | **Стани оплат — окремі сутності** | у `scheda cliente` `pagato` / `da pagare` / `da fare` мають різний вигляд (колір/іконка/секція), не один список зі статус-чіпом |

### 4.5. Перетікання між режимами

Кожна дихотомія керується **одним вивченим жестом** — segment-slide із морфінгом
каркаса: спільне лишається на місці, відмінне перетікає.

**`svolto` ↔ `programmato`** (на `Crea` і всюди, де є тип запису):

| | `svolto` | `programmato` |
|---|---|---|
| палітра | тепла/нейтральна, суцільні рамки | холодна/пунктирна, «attesa» |
| акцент-дія | `INCASSA` | `PIANIFICA` / `DATA` |
| іконіка | ✓ галочка | ◷ годинник/календар |

- Перехід: спільні поля (`cliente`/`operaio`/`modalità`) **залишаються** на місці
  (shared-element), cross-fade лише оформлення; поля-відмінності
  (`importo` ↔ `data pianificata`) — height-морф.
- Нативне перетікання в житті: `programmato`, що виконано → дія
  `«segna come svolto»` прямо на картці → пунктир ущільнюється в суцільне, картка
  впадає в потік `svolto` тієї дати. Той самий `lavoro`, інший стан.

**`da incassare` ↔ `da pagare`** (у `Soldi`): той самий segment-slide; список
зберігає геометрію рядка, змінюються лише напрям стрілки потоку (▲/▼) і палітра
(вхідне/зелене ↔ вихідне/бурштин). Каркас (header, FAB, бейдж Σ) лишається,
підсумок перераховується count-up.

**`Soldi` → `Dashboard`:** tap по блоку → zoom-expand блоку на весь екран → стає
Dashboard-секцією; назад — зворотний zoom у той самий блок («провалився в цифру»).

**`Agenda` → `Dettaglio lavoro`:** картка піднімається (lift) і розкривається у
повний екран; `codice` і назва клієнта **летять** shared-element-ом у шапку.

Спільний принцип: спільне лишається, відмінне морфить → відчуття одного
застосунку з багатьма станами, а не «купи екранів».

### 4.6. Головний шлях vs рідкісні сценарії

```
ВХІД ▶ Agenda (дім) — одразу місяць зроблених робіт.
        │
ГОЛОВНИЙ ШЛЯХ (≈80%, соло-`svolto`, 3-4 тапи):
  ( + ) → svolto → «io» → modalità → salva → картка в Agenda
        ▼
  картка → «incassa» (інлайн, повна сума) → готово.

РІДКІСНІ СЦЕНАРІЇ (нативно розгортаються, не захаращують головний шлях):
  • клієнта нема       → модалка «imposta tariffa» (лише при потребі)
  • вибрати наявного   → екран-список → назад з усіма налаштуваннями
  • працював із кимось → +operaio розкриває [mie ore | sue ore]
  • spese для нетто    → тригер-рядок → секція
  • план на майбутнє   → перемкнути на programmato (+ giornate для багатьох днів)
  • «Incassa subito»   → з Soldi/картки: вся оплата одразу, БЕЗ годин
                         (для чистого preventivo)
```

Принцип: головний шлях **ніколи не платить складністю** за рідкісні випадки —
вони живуть у згорнутих гілках і окремих екранах.

### 4.7. Глобальна дія «створити запис»

```
┌─────────────────────────────────────────────────────────┐
│  FAB «+» — постійно по ЦЕНТРУ таб-бара, на всіх екранах.  │
│  Єдина точка входу в «Crea registrazione».               │
└─────────────────────────────────────────────────────────┘
```

- **Де живе:** центральний FAB у таб-барі, присутній на всіх трьох головних
  екранах (`Agenda` / `Soldi` / `Anagrafiche`). Єдина глобальна дія, нуль
  дублюючих кнопок «створити» по кутах.
- **Як викликається:** tap → bottom-sheet `«Crea registrazione»`, що морфить із
  самого FAB (continuity), а не «чужий» екран. Дефолт: `[ svolto ]` + `«io»`.
  Перше й єдине питання — розвилка `svolto`/`programmato`.
- **Контекстне насіння (seeding):** екран той самий, контекст ловиться з місця
  виклику:

| Контекст виклику | Передзаповнення |
|---|---|
| з `Agenda` (день обрано) | дата запису = той день |
| зі `scheda cliente` | `cliente` підставлено |
| зі `scheda operaio` | `operaio` підставлено, одразу питання `svolto`/`programmato` |
| з `Soldi` | схил до `svolto` (контекст грошей) |

### 4.8. Канонічні рішення та межі v1

- **Канон модальностей розрахунку — чотири:** `preventivo`, `ore`, `giornate`,
  `totale`. Старий код містив `preventivo`/`ore`/`misto`; у новій моделі `misto`
  поглинається комбінацією `giornate`/`totale` + кілька `preventivo`.
- **`codice parlante`** генерується автоматично з ініціалів клієнта (напр. `MR`,
  `MR1`) при створенні — не вводиться вручну.
- **Поза v1 (явно):**
  - окремий екран-таб для `Dashboard` (лишається провалом, не пунктом навігації);
  - складні інлайн-дії на картці зі складним сценарієм виносяться на екран —
    інлайн лишається лише для простих (повна сума + швидкі суми);
  - будь-яка дублююча навігація до одного екрана понад два канонічні входи.


## 5. Екрани «Agenda» і «Dashboard»

`Agenda` — центр світу і стартова точка (дім у таб-барі): календар-щоденник за **весь місяць**, вертикальний потік компактних карток робіт. `Dashboard` — споглядальний екран аналітики **поза таб-баром**, що показує дані ОКРЕМО по `clienti` та ОКРЕМО по `operai`, ніколи не змішуючи вхід і вихід грошей.

Канонічний словник:

| UA | IT (продукт) | Значення |
|---|---|---|
| зроблено | `svolto` | `faseLavoro(l) === "fatto"` — минуле, гроші, квитанція |
| заплановано | `programmato` | `faseLavoro(l) === "da_fare"` — майбутнє, намір, креслення |
| бирка дати | `targhetta` | плаваюча латунна бирка (Telegram-стиль) |
| створити запис | «`Crea registrazione`» | глобальна дія (FAB) |

### 5.1 `Agenda` — композиція

```
┌───────────────────────────────────────────────────┐
│ ░peek Dashboard          peek Anagrafiche░         │
│  ШАПКА (statica, скролиться геть)                  │
│  «Agenda»          [◀ mese]  Giugno 2026  [mese ▶] │
│  ╭───────────────────╮                             │
│  │  ⬢  GIO 5 giugno  │  ← targhetta (sticky)        │
│  ╰───────────────────╯                             │
├───────────────────────────────────────────────────┤
│ ▼ ПОТІК МІСЯЦЯ (scroll)                            │
│  ── giovedì 5 ──────────────────────  €128 · 6h    │
│   ┌─ card svolto ───────────────────┐              │
│   ┌─ card svolto ───────────────────┐              │
│  ── venerdì 6 ──────────────────────  €0 · план    │
│   ┌ ─ card programmato (пунктир) ─ ┐               │
│  ── sabato 7 ───────────────────────               │
│   ···· nessun lavoro · [ + crea ] ····             │
├───────────────────────────────────────────────────┤
│ ТАБ-БАР: [Agenda🏠]   ＋   [Soldi💶][Anagr👥]      │
└───────────────────────────────────────────────────┘
```

Три вертикальні зони: **шапка** (`statica`, назва + перемикач місяця, скролиться геть) → **`targhetta`** (sticky, завжди видима) → **потік місяця** (секції днів).

**Які дні рендеримо** (РІШЕННЯ — не всі 30 порожніх поспіль):
- кожен день із хоча б однією роботою (`svolto` або `programmato`);
- `oggi` завжди, навіть порожній (якір);
- суцільний діапазон порожніх днів → один роздільник «`···· 8–11 giu · nessun lavoro ····`» (тап розгортає поштучно).

**Заголовок дня** — `dataLungaIT` дає `weekday`+число; праворуч денний підсумок Σ`lordo` · Σ`oreTotali` («`€128 · 6h`»). Сьогодні — акцент `ottone` + мітка «`oggi`». Картки впорядковані за `l.ordineNelGiorno`, далі `l.oraInizio`; `svolto` і `programmato` лежать у хронології впереміш, відмінні лише візуально.

### 5.2 Компактна картка роботи

```
┌─────────────────────────────────────────────────────┐
│  MR-03-12-05   Mario Rossi              [≡ svolto]   │
│  Potatura olivi                          14:00–17:00 │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │  ← перфорація
│  6h        ✓ €90 incassato      ◷ €38 da incassare   │
│  operai:  ⬡io  ⬡Luca                                 │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
│            [ Incassa ]            [ ⋯ ]               │
└─────────────────────────────────────────────────────┘
```

Усе — деривати з `calcoloLavoro(db, l)` + `db.clienti`, жодних збережених агрегатів. **Компактність = три якорі** (РІШЕННЯ): `ore`, `incassato`, `daIncassare`. `lordo`/`netto`/`costoCollaboratori`/`spese`/історія — лише на деталі (`Cantiere`).

| Елемент | Джерело |
|---|---|
| `codice` штамп | `codiceCliente(db, l.clienteId)` |
| ім'я клієнта | `db.clienti[l.clienteId]` |
| штамп фази | `faseLavoro(l)` → «`svolto`»/«`programmato`» |
| `titolo`, смуга часу | `l.titolo`, `l.oraInizio–oraFine` |
| **години** | `calc.oreTotali` → `ore(n)` |
| **оплачено** | `calc.incassato` → «€90 incassato» |
| **очікую** | `calc.daIncassare` → «€38 da incassare» |
| chip-операї | `calc.partecipanti[]`, «`io`» першим |

**Грошовий рядок керується `statoIncasso`** (поріг `0.005`):
- `non_pagato` → лише «`◷ €X da incassare`» (тон `attenzione`);
- `parziale` → обидва: «`✓ €90 incassato · ◷ €38 da incassare`»;
- `pagato` → лише «`✓ €128 incassato`» зі штампом-печаткою (тон `positivo`).

**Chip операя `⬡`:** завжди «`io`» першим (`operatoreIo(db)`), далі `partecipanti`; тап → провал у `OperatoreScheda`; >3 → «`io · Luca +2`».

### 5.3 `svolto` vs `programmato` — візуальна відмінність

Спільний каркас, різний характер (квитанція vs креслення):

```
svolto — «квитанція»                programmato — «креслення»
┌═════════════════════════════┐     ┌ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┐
║ суцільна тепла рамка        ║     ╎ ПУНКТИРНА прохолодна рамка╎
║ carta-svolto (тютюн)        ║     ╎ carta-programmato (синька)╎
║ штамп ✓ SVOLTO під кутом    ║     ╎ мітка ◷ (годинник)        ╎
║ повні суми, моноширинні     ║     ╎ «previsto» замість сум    ╎
║ майже без тіні (лягло)      ║     ╎ висока мʼяка тінь (плаває)╎
║ дія: [ Incassa ]            ║     ╎ дія: [ Segna svolto ]     ╎
└═════════════════════════════┘     └ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┘
```

| Вісь | `svolto` | `programmato` |
|---|---|---|
| Рамка | суцільна | пунктирна |
| Поверхня | `carta-svolto` (тепла) | `carta-programmato` (прохолодна) |
| Штамп | «`svolto`» ✓ під кутом | мітка «`programmato`» ◷ |
| Тінь (=час) | майже нема (лягло) | висока, мʼяка (плаває) |
| Гроші | `incassato`/`daIncassare` | сум немає; `previsto` |
| Головна дія | «`Incassa`» | «`Segna svolto`» |

**Канон `programmato` і гроші:** за фазою `da_fare` домен дає `incassato = 0`, тому замість оплат показуємо очікуваний обсяг: `modo === "preventivo"` → «`€X previsto`» (`l.prezzo`); `modo === "ore"` → план годин/тариф, якщо заданий `durataPrevistaOre`.

**Морфінг «`Segna svolto`»:** пунктир→суцільна, прохолодна→тепла, ◷→✓, картка «осідає» (тінь падає), лишається в тому ж дні; доменно `l.fase → "fatto"`.

### 5.4 Дії на картці (контекстні до `fase` + `statoIncasso`)

**`svolto`:**
```
non_pagato / parziale → [ Incassa ]    (латунна, головна)
pagato                → [ ✓ Saldato ]  (disabled-печатка)
⋯ : Apri lavoro · Modifica · Aggiungi spesa · Duplica · Elimina
```

«`Incassa`» — **інлайн швидка дія**, розгортається в картці (не окремий екран):
```
┌─────────────────────────────────────────────────┐
│  Incassa da Mario Rossi                          │
│  da incassare:  € 38,00                          │
│  [ tutto € 38 ]  [ € 20 ]  [ altro… ]            │
│  ☐ Incassa subito (intero preventivo, senza ore) │
│            [ Conferma incasso ]                  │
└─────────────────────────────────────────────────┘
```
Доменно: `pagamentoApertoLavoro(db, l.id)`, `importoIncassato = importoAtteso`, `dataIncasso = oggi` → `statoIncasso → "pagato"`, рух у `movimenti` (`incasso +1`). Чекбокс «`Incassa subito`» активний лише для `modo === "preventivo"` — інкасує весь `prezzo` без прив'язки до годин.

**`programmato`:**
```
головна → [ Segna svolto ]   (морфінг, §5.3)
⋯ : Apri lavoro · Riprogramma (дата) · Modifica · Elimina
```

**Спільне:** тап по тілу (поза кнопками/chip) → провал у `Cantiere` (повна ієрархія: періоди, `per_giorni`, історія оплат, `spese`, `netto`); `codice`+ім'я летять shared-element-ом.

### 5.5 `targhetta` — плаваюча бирка дати (sticky, Telegram-стиль)

Латунний обʼємний жетон із `weekday`+число+місяць, що плаває згори і **при скролі змінюється на день, який наближається**.

```
СКРОЛ ВНИЗ:
 [targhetta: GIO 5]      [targhetta: GIO 5]       [targhetta: VEN 6]
 ── giovedì 5 ──         ── giovedì 5 ──(вгору)    ── venerdì 6 ──
  card svolto             card svolto               card programmato
 ── venerdì 6 ──         ── venerdì 6 ──(підпирає)  ...
  крок 1: бачимо 5       крок 2: 6 підпирає 5       крок 3: targhetta = 6
```

Поведінка:
1. **Прилипання** — поки в секції дня, бирка показує цей день.
2. **Push-out** — кожен заголовок дня `position: sticky; top: 0` під `targhetta`; коли наступний sticky-заголовок штовхає попередній, бирка синхронно крос-фейдить число (класична Telegram date-bubble на латунному жетоні).
3. **Тап** → компактний місячний пікер (мінікалендар), стрибок на будь-яку дату.
4. **Межі** — на верху перший непорожній день, на низу останній; перехід через місяць — лише кнопками шапки.
5. **Рух** — спокійний крос-фейд («перегортання сторінки»), легке зміщення 2-4px; `prefers-reduced-motion` → миттєва зміна без зсуву.

### 5.6 Навігація в часі та порожні стани

**Перемикач місяця** (РІШЕННЯ — час кнопками, простір свайпом):
```
[◀]   Giugno 2026   [▶]   ← meseAnnoIT(anno, mese)
```
`[◀]`/`[▶]` = горизонтальний слайд потоку. Якщо відкритий не поточний місяць — латунна мітка «`oggi`» повертає на сьогодні. Гориз. свайп ←/→ зарезервований під просторову мапу (`Dashboard ⇄ Agenda ⇄ Anagrafiche`), тому зміна місяця — **лише кнопками**.

**Порожні стани:**
```
Порожній день:   ···· nessun lavoro · [ + crea ] ····
Прогалина:       ···· 8–11 giu · nessun lavoro ····  (тап розгортає)
```
```
Порожній місяць:
┌───────────────────────────────────────────────┐
│           🌳  Nessun lavoro in Giugno 2026      │
│   Tocca ＋ per creare la prima registrazione    │
│             [ + Crea registrazione ]            │
│        ◀ Maggio              Luglio ▶           │
└───────────────────────────────────────────────┘
```
«`+ crea`» / FAB відкриває «`Crea registrazione`» з передзаповненою `data` (активний день із `targhetta` або обраний тапом по заголовку), дефолт сегмент «`svolto`» + операйо «`io`».

**Скрол:** шапка `statica` (геть); `targhetta` sticky; заголовки днів sticky під нею (push-out); `svolto` нерухомі, `programmato` злегка плавають; інерція «паперова», без bounce. Віртуалізація довгого потоку — закладена, **поза v1-візуалом**.

### 5.7 `Dashboard` — композиція і режими

Входи (РІШЕННЯ — свідомо НЕ в таб-барі): (1) свайп вліво з `Agenda`; (2) тап по підсумковому блоку в `Soldi`. Ключ: дані **ОКРЕМО** `Clienti` (вхід грошей) і **ОКРЕМО** `Operai` (вихід), ніколи разом.

```
┌───────────────────────────────────────────────────┐
│  «Dashboard»                              [ ✕ ]   │  ← ✕/свайп вправо = на Agenda
│      [ Clienti ]      [ Operai ]                  │  ← сегмент-slide (sticky)
│  [ Periodo: Giugno ▾ ]  [ Stato ▾ ]               │  ← фільтри (sticky)
│  ┌─ ПІДСУМКОВА СТРІЧКА (3-4 KPI режиму) ─────────┐ │  ← скролиться геть
│ ───────────────────────────────────────────────── │
│  СПИСОК СУТНОСТЕЙ (клієнти АБО операї)             │
│   ┌─ рядок ─┐ ┌─ рядок ─┐ ┌─ рядок ─┐  (тап→провал)│
└───────────────────────────────────────────────────┘
```

Перемикання `Clienti ⇄ Operai` — slide: каркас списку не перебудовується, лише перефарбовується і змінює субʼєктів/напрям. Палітра: `Clienti` тепла «до мене» (`positivo`); `Operai` бурштин «від мене» (`uscita`).

### 5.10 Фільтри, порожні стани, скрол `Dashboard`

**Фільтри** (Radix bottom-sheet/поповер, не постійні панелі; активний = латунний чип із «`✕`»):

| Фільтр | Значення |
|---|---|
| `Periodo` | «`Questo mese`» (дефолт) · «`Mese precedente`» · «`Quest'anno`» · «`Tutto`» |
| `Stato` (Clienti) | «`Tutti`» · «`Da incassare`» · «`In ritardo`» · «`Saldati`» |
| `Stato` (Operai) | «`Tutti`» · «`Da pagare`» · «`Parziali`» · «`Saldati`» |

Період фільтрує: `Clienti` — за датами робіт/платежів; `Operai` — `periodo` у `libroOperatore(db, id, "YYYY-MM")`. «`Personalizzato`» (`dal`/`al`) — **поза v1**.

**Порожні стани** (порожнеча = добра новина, тон `positivo`):
```
Clienti:  📄 Nessun cliente con movimenti in Giugno   [ Cambia periodo ][ Tutto ]
Operai:   👥 Nessuna ora registrata in Giugno         [ Cambia periodo ]
In ritardo порожній:  ✓ Nessun cliente in ritardo — tutto in regola
```

**Скрол:** шапка + сегмент `Clienti/Operai` — `sticky` (на відміну від `Agenda`); рядок фільтрів sticky під шапкою; KPI-стрічка скролиться геть (це «шапка контенту»); список звичайний потік без sticky-днів (реєстр, не календар); перемикання режиму зберігає позицію зверху.

### 5.11 Узгодження каркаса `Agenda` ↔ `Dashboard`

| Аспект | `Agenda` (дім) | `Dashboard` (поза навігацією) |
|---|---|---|
| Вхід | таб-бар, стартовий | свайп вліво з `Agenda` АБО блок у `Soldi` |
| Sticky-герой | `targhetta` | сегмент `Clienti/Operai` + фільтри |
| Шапка | `statica` (геть) | `sticky` (перемикач завжди) |
| Розкладка | потік днів за місяць | список сутностей (реєстр) |
| Дихотомія/жест | `svolto / programmato` (у потоці) | `Clienti / Operai` (segment-slide) |
| Одиниця | картка **роботи** (`Lavoro`) | рядок **сутності** (`Cliente`/`Operatore`) |
| Провал | у `Cantiere` | у `ClienteScheda`/`OperatoreScheda` |
| Гроші | `ore`/`incassato`/`daIncassare` | KPI + борг по сутності |
| Змішування вх/вих | — | **заборонено** (Clienti ≠ Operai) |
| Реджест-мап | свайп = простір; час кнопками | свайп вправо = назад на `Agenda` |

**Поза v1 (явно):** `Personalizzato` період; drag-and-drop `programmato` між днями (v1 — «`Riprogramma`»); графіки/тренди в `Dashboard` (v1 текстово-реєстровий); віртуалізація потоку `Agenda`; точний склад денного підсумку (дефолт `lordo`+`ore`).


## 6. Створення роботи — «svolto» і «programmato»

Один екран `Crea registrazione`, що **росте знизу**. Перший дотик задає
**характер** запису сегментом `[ svolto | programmato ]`; усе нижче
розгортається нативно з того ж місця. Соло-`svolto` проходиться за 3-4 тапи,
уся глибина (періоди, рядки днів, другий operaio, `spese`, тариф, конверсія
плану) — за прогресивним розкриттям, не захаращуючи головний шлях.

Канонічний наслідок перемикача:

| Сегмент | `fase` (домен) | Характер | CTA | Гроші-на-вхід |
|---|---|---|---|---|
| `svolto` | `"fatto"` | «квитанція»: тепла, суцільна рамка, штамп `SVOLTO`, осіла | `Salva registrazione` (латунь) | є — `Già incassato?` |
| `programmato` | `"da_fare"` | «креслення олівцем»: синька, пунктир, висока тінь, плаває | `Pianifica` (графіт) | **немає** (`incassato≡0`) |

> Канон: `fase:"da_fare"` ⇒ `calcoloLavoro.incassato = 0` примусово. Тому
> інкасо-секція має сенс **лише** у `svolto`; план грошей не приймає.

---

### 6.1 Канонічна модель запису

```
Lavoro {
  fase: "fatto" | "da_fare",            // ← сегмент svolto/programmato
  modo: "preventivo" | "ore",           // вісь компенсації
  conteggio: "totale" | "per_giorni",   // як рахуються години (тільки modo="ore")
  clienteId | null,                     // null → модалка tariffa (§6.4)
  titolo, data,                         // data з контексту або сьогодні
  periodo?: { dal, al } | null,         // мітка-діапазон (preventivo/totale)
  prezzo?,                              // тільки modo="preventivo"
  tariffaClienteSnapshot?,              // заморожений €/год (тільки modo="ore")
  partecipanti: PartecipanteLavoro[],   // мінімум «io»
  contaMieOreComeCosto: boolean,        // default false
}
```

Побічні записи (бо `db.ore` — єдине джерело істини годин, `db.spese` — витрат):

```
RegistrazioneOre[]  // один на (день × operaio) — лише у svolto / при конверсії
Spesa[]             // нуль або більше, прив'язані до lavoroId
Pagamento?          // лише у svolto; preventivo: atteso=prezzo, ore: atteso=oreCli×tariffa
```

**4 «modalità di calcolo» на фасаді = 2 ортогональні осі домену:**

| Ярлик | `modo` | `conteggio` | Ввід годин |
|---|---|---|---|
| `preventivo` | `preventivo` | — | немає (тільки `prezzo`) |
| `ore` | `ore` | `totale` | одна сума, прив'язана до `data` |
| `giornate` | `ore` | `per_giorni` | рядки `{data, ore}` |
| `totale` | `ore` | `totale` | одна сума за `periodo` (дні неважливі) |

`ore` і `totale` мають однаковий канон; різниця суто UX — `ore` = «сьогодні
відпрацював N», `totale` = «за період накопичилось N, дні неважливі» (тому
`totale` відкриває `periodo`). Канон їх не розрізняє — розрізняє форма вводу.

### 6.2 Вхід і миттєвий вибір характеру

```
ВХІД ▶ Agenda (дім) → тап FAB «＋» (центр таб-бара)
        │  bottom-sheet «Crea registrazione» морфить із FAB
        ▼
┌───────────────────────────────────────────────┐
│  ╔═══════════╗                                 │
│  ║ ✓ svolto  ║   ◷ programmato     [штамп ...] │  ← segment-slide
│  ╚═══════════╝                                 │
│   (тепле,активне)   (синька,прохолодне)        │
└───────────────────────────────────────────────┘
```

**Контекстне передзаповнення** (екран той самий, контекст із місця виклику):

| Звідки `＋` | Передзаповнено |
|---|---|
| `Agenda`, обрано день | `data` = той день |
| `Agenda`, день не обрано | `data` = сьогодні |
| картка `cliente` | `clienteId`, рядок `Cliente` згорнутий-заповнений |
| картка `operaio` | той operaio у `partecipanti` |

**Перехід `svolto ⇄ programmato` — морфінг каркаса, не зміна екрана:**

- **Спільний хребет лишається на місці:** `Cliente` · `Operaio "io"` ·
  `Modalità di calcolo`. Лише cross-fade оформлення (рамка тепла↔синька,
  ✓↔◷).
- **`svolto`:** з'являється рядок `Già incassato?`; числа реальні; мікрокопі
  минулого часу («Ho lavorato», «Già incassato?»).
- **`programmato`:** замість грошей-на-вхід виростає блок `Quando`
  (дата/період) + тригер `Operai pianificati`; числа очікувані (`previsto` /
  `stima ≈`); мікрокопі майбутнього («Pianifico», «Quando»). **Вся секція
  грошей зникає.**

### 6.3 Прогресивне розкриття — кістяк, що росте знизу

Закон: **«порожнє поле не існує»**.

```
┌─────────────────────────────────────────────────────────┐
│  ╔═══════════╗   ◷ programmato            [штамп SVOLTO] │
│  ║ ✓ svolto  ║                                           │
│  ╚═══════════╝                                           │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
│  Data ........................... ven 23 giu 2026  ▾     │  ← з контексту
│  Cliente ...................... + scegli o nuovo   ›     │  ← тригер-рядок
│  Operaio ........................... «io»          ▾     │  ← дефолт прихований
│  Modalità di calcolo                                     │
│  ┌──────────┬───────┬──────────┬────────┐                │
│  │preventivo│  ore  │ giornate │ totale │                │  ← ЄДИНА розвилка
│  └──────────┴───────┴──────────┴────────┘                │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
│  + Ho speso qualcosa (spese)                       ›     │  ← тригер
│  + Ho lavorato con qualcuno                        ›     │  ← тригер
│            [  Salva registrazione  ]   ← латунна CTA     │
└─────────────────────────────────────────────────────────┘
```

Поки `Modalità` не обрано — нижче нічого немає; `Salva` неактивна (§6.9).
Патерни: тригер-рядок замість поля; `Modalità` — єдина розвилка низу;
`«io»` ховає соло-сценарій в один рядок; модалка — лише для рідкісного.

### 6.4 Вибір клієнта (push-список зі станом) + кейс без клієнта

**Тригер `Cliente …… + scegli o nuovo ›` → окремий екран-push** (вибір частий,
заслуговує простору + пошуку):

```
ЕКРАН «Scegli cliente»  (push, shared-element: рядок летить у шапку)
┌─────────────────────────────────────────────────────────┐
│  ‹ Indietro                          Scegli cliente      │
│  🔍 Cerca per nome o codice…                             │
│ ─────────────────────────────────────────────────────── │
│  MR-03-12-05   Mario Rossi          Portoferraio    ›   │
│  MR1-00-09-00  Maria Ricci          Capoliveri      ›   │
│ ─────────────────────────────────────────────────────── │
│  + Nuovo cliente                                    ›   │
└─────────────────────────────────────────────────────────┘
```

- `codice parlante` (`II-GG-SS-AA`) обчислюється на льоту через `codiceCliente`;
  пошук по `nome`/`cognome`/`inizialiCodice`.
- Тап по рядку → **повернення на `Crea` з УСІМА налаштуваннями збереженими**.
  Рядок стає `Cliente …… MR-03-12-05 · Mario Rossi  ✎`.

> **Канон збереження стану (критично):** стан `Crea` живе у **draft-чернетці
> у Zustand-сторі**, не в локальному стейті компонента. Push на список і
> повернення **не розмонтовує** чернетку — modalità, години, рядки днів,
> `spese`, перемикачі лишаються. Те саме для push-списку operai (§6.6).

`+ Nuovo cliente` → міні-форма (`nome`, `cognome`, `luogo?`, `tariffaOraria?`,
`telefono?`); `inizialiCodice` присвоюється через
`assegnaIniziali(nome, cognome, esistenti)` (третій «Mario R…» → `MR2`);
новий клієнт одразу вибирається.

**Кейс БЕЗ клієнта → модалка тарифу.** Збереження без клієнта дозволено
(швидкий запис / разова робота). Поведінка залежить від modalità:

```
┌──────────────── modale «Imposta tariffa» ────────────────┐
│  Nessun cliente collegato.                                │
│  Quanto vale un'ora di questo lavoro?                     │
│     € [   28,00   ] / ora                                 │
│  ☐ Salva senza tariffa (0 €/ora)                          │
│      [ Annulla ]      [ Conferma ]                        │
└───────────────────────────────────────────────────────────┘
```

- `modo:"ore"` (ore/giornate/totale): без клієнта тарифу немає → модалка при
  `Salva`. Сума → `tariffaClienteSnapshot`; «senza tariffa» → `=0` → `lordo=0`,
  години фіксуються, інкасо-секція ховається.
- `modo:"preventivo"`: тариф не потрібен (`lordo=prezzo`) → модалка **не
  з'являється**.

> **Рішення:** робота без клієнта канонічно дозволена; `clienteId` опціональний;
> у картці — «Senza cliente» з можливістю прив'язати пізніше.

### 6.5 4 режими розрахунку — поля, валідації, формули

`arrotonda(n)` = округлення до 2 знаків; поріг грошей `0.005`.

**`preventivo`** — `lordo = arrotonda(prezzo ?? 0)`, годин немає:

```
  Importo concordato    € [  800,00  ]              ← prezzo
  Periodo (facoltativo)   dal [10 giu] al [14 giu] ✕ ← мітка-діапазон
```

| Поле | Канон | Тип | Обов'язк. |
|---|---|---|---|
| Importo concordato | `prezzo` | € ≥ 0 | **так** (мінімум для Salva) |
| Periodo | `periodo:{dal,al}\|null` | 2 ISO-дати, `dal≤al` | ні |

`periodo` **не впливає** на гроші. `prezzo=0` дозволено лише з попередженням.

**`ore`** — `lordo = arrotonda(oreCliente × tariffaCli)`, одна `data`:

```
  Quante ore ho lavorato?   [  6,5  ] h             ← ore
  Tariffa cliente   28,00 €/h (da Mario Rossi)  [✎] ← tariffaClienteSnapshot
  Stima lordo       6,5 × 28,00 = 182,00 €          ← live read-only
```

| Поле | Канон | Тип | Обов'язк. |
|---|---|---|---|
| Quante ore | `RegistrazioneOre.ore` для (`data`,`io`) | ≥ 0, крок 0,5 | **так** |
| Tariffa cliente | `tariffaClienteSnapshot` | €/год ≥ 0 | авто з клієнта; редаговане |

Тариф: з `cliente.tariffaOraria` (заморожується), або модалка (§6.4); `✎` =
разовий тариф без зміни клієнта. Валідація: `ore>0` (інакше блок «Inserisci
almeno qualche ora»); `tariffa=0` → попередження «Lordo sarà 0 €».
Побічний запис: `RegistrazioneOre{clienteId, lavoroId, operatoreId:io, data, ore}`.

**`giornate`** — рядки `{data, ore}`, `lordo = arrotonda(Σore × tariffaCli)`:

```
  Giornate di lavoro
  ┌────────────┬────────┬───┐
  │ lun 10 giu │ 8,0 h  │ ✕ │     ← RigaGiorno
  │ mar 11 giu │ 6,5 h  │ ✕ │
  └────────────┴────────┴───┘
  + Aggiungi giornata
  Totale ore  18,5 h  ·  Tariffa 28 €/h [✎]  ·  Stima 518,00 €
```

`+ Aggiungi giornata`: дата = остання +1 день, фокус у поле годин. `✕` видаляє
рядок; останній не зникає (порожній рядок-заготовка). Дублікат дати дозволено
(сумується). Побічно: один `RegistrazioneOre` на кожен рядок.

**`totale`** — одна сума за період, `lordo = arrotonda(oreTotali × tariffaCli)`:

```
  Ore totali (senza giorni)  [ 18,5 ] h
  Periodo (facoltativo)   dal [10 giu] al [14 giu] ✕
  Tariffa 28 €/h [✎]  ·  Stima 518,00 €
```

Побічно: один `RegistrazioneOre{data: periodo.al ?? data, operatoreId:io, ore}`.

> Канон: фактичні години для розрахунку **завжди з `db.ore`**, тому всі режими
> матеріалізують години в `RegistrazioneOre` (Dashboard/профілі узгоджені).

### 6.6 `Spese` — для чистого (`netto`)

Тригер `+ Ho speso qualcosa`; доки не тапнуто — секції нема.

```
  Spese di questo lavoro
  ┌───────────┬──────────┬────────┬───┐
  │ Benzina   │ Trasporto│ 35,00 €│ ✕ │   ← Spesa{categoria,descrizione?,importo}
  │ Materiali │ Concime  │ 85,00 €│ ✕ │
  └───────────┴──────────┴────────┴───┘
  + Aggiungi spesa   ·   Totale spese  120,00 €
```

| Поле | Канон | Тип | Обов'язк. |
|---|---|---|---|
| categoria | `CategoriaSpesa` | benzina/materiali/attrezzi/altro | так (дефолт materiali) |
| descrizione | `descrizione?` | текст | ні |
| importo | `importo` | € > 0 | так |

`Spesa{categoria, importo, data:data-запису, clienteId, lavoroId}`.
`speseTotali = arrotonda(Σ db.spese[lavoroId].importo)`.

```
netto = arrotonda(lordo − speseTotali − costoCollaboratori)
```

Spese зменшують **чистий**, не зменшують `lordo` і не впливають на `daIncassare`.

### 6.7 Співпраця: мої години vs їхні години

Тригер `+ Ho lavorato con qualcuno`; доти `partecipanti` = тільки «io» (форма
соло). Тап → `Chi ha lavorato` + `+ Aggiungi operaio` → **push-список operai**
зі збереженням стану (як §6.4). Вибір додає
`PartecipanteLavoro{collaboratoreId, tariffaSnapshot}` — **`tariffaSnapshot`
заморожується** в момент додавання (пізніша зміна тарифу не перерахує роботу).
`+ Nuovo operaio` → міні-форма (`nome`, `ruolo`, `tariffaOraria?`).

**Ввід годин залежить від modalità:**

```
ore/totale (conteggio:"totale")          giornate (conteggio:"per_giorni")
┌──────────────────────────┐            ┌────────────┬──────┬──────┬───┐
│ «io» (Marco)  [ 6,5 ] h  │            │            │  io  │ Luca │   │
│ Luca·16€/h    [ 6,0 ] h ✕│            │ lun 10 giu │[8,0] │[8,0] │ ✕ │
└──────────────────────────┘            │ mar 11 giu │[6,5] │[4,0] │ ✕ │
  Ore totali  12,5 h                    └────────────┴──────┴──────┴───┘
                                         Totale: io 14,5 · Luca 12,0 · tutti 26,5
```

> `preventivo`: `lordo=prezzo` (від годин не залежить), але НЕ-«io» учасники
> **все одно вводять свої години** — вони не йдуть у `lordo`, а йдуть у
> `costoCollaboratori = Σ(ore × tariffaSnapshot)` для не-io → зменшують `netto`.

**`contaMieOreComeCosto`** — перемикач у секції «Chi ha lavorato»:

```
  ☐ Conta le mie ore come costo
     (di default le mie ore sono guadagno, non spesa)
```

- `false` (дефолт): `costo «io» = 0` → мій час = чистий прибуток.
- `true`: `costo «io» = arrotonda(ore_io × tariffaSnapshot_io)` → у
  `costoCollaboratori`, зменшує `netto`. Виключення стосується **лише**
  учасника з `collaboratoreId === ioId`.

**Приклад (`ore`, io 6,5h, Luca 6h@16, клієнт 28€/h, spese 0):**

```
lordo = arrotonda(12,5 × 28) = 350,00 €
costo io = 0  (contaMieOreComeCosto=false)
costo Luca = arrotonda(6 × 16) = 96,00 €
netto = arrotonda(350 − 0 − 96) = 254,00 €
```

> У seed `titolare` має `tariffaOraria: null` → `0`, тож `contaMieOreComeCosto`
> дає ефект лише коли «io» отримав ненульовий тариф.

Побічно: `RegistrazioneOre` на кожного учасника × день/тоталь →
живить `libroOperatore(Luca)` → `dovuto = 96 €` у `Soldi → da pagare`.

### 6.8 `Già incassato?` / `Incassa subito` (тільки svolto)

```
  Già incassato?
   ◉ No, da incassare              ← дефолт
   ○ Sì, tutto incassato oggi      ← «Incassa subito»
   ○ In parte:  € [        ]
```

При `Salva` створюється `Pagamento` (origine `preventivo`/`ore` за `modo`):

| Вибір | `importoAtteso` | `importoIncassato` | `dataIncasso` | `statoIncasso` |
|---|---|---|---|---|
| No, da incassare | `lordo` | `0` | — | `non_pagato` |
| Sì, tutto oggi | `lordo` | `lordo` | сьогодні | `pagato` |
| In parte € X | `lordo` | `X` (0<X<lordo) | сьогодні | `parziale` |

`Incassa subito` закриває цикл «зробив → узяв гроші» в один екран: платіж одразу
у `movimenti` (+1) та `riepilogoMese.incassato`. Якщо `lordo=0` — секція
ховається. `In parte` обмежено `X ≤ lordo` (`daIncassare` ніколи не від'ємний).

### 6.9 Збереження `svolto` — потік і стани

```
1. Salva (clienteId=∅, modo=ore) → модалка Imposta tariffa → Conferma
2. валідація: ore>0, tariffa≥0, рядки/періоди коректні (§6.10)
3. створення (атомарно у store):
     Lavoro{ fase:"fatto", modo, conteggio, clienteId, data,
             tariffaClienteSnapshot, partecipanti, contaMieOreComeCosto }
     RegistrazioneOre × учасник × день
     Spesa × рядок
     Pagamento (за вибором «Già incassato?»)
4. sheet закривається з хаптик-«штампом SVOLTO»; картка «лягає» в Agenda на
   дату data (мінімальний рух, минуле не тремтить); shared-element: cliente+
   codice летять у картку. Глибина (періоди, giornate, історія оплат, netto)
   — за провалом у роботу.
```

### 6.10 Крайові випадки

| Випадок | Рішення |
|---|---|
| **Нуль годин** (ore/giornate/totale = 0) | `Salva` блок; «Inserisci almeno qualche ora» |
| **Відсутній тариф** (modo:ore, без клієнта/тарифу) | Модалка `Imposta tariffa`; «senza tariffa» → `0` |
| **Видалення останнього рядка giornate** | Лишається порожній рядок-заготовка; зберегти без дня не можна |
| **Видалення operaio** | `✕` прибирає учасника; «io» видалити **не можна** |
| **Зміна modalità після вводу** | Поля ховаються, значення тримаються в draft по-режимно; неоднозначне перенесення → тост «Alcuni dati non si applicano», без втрати |
| **`prezzo=0` preventivo** | Дозволено з попередженням; `lordo=0`; інкасо ховається |
| **`periodo` `dal>al`** | Міняємо місцями або блок «La data 'dal' deve precedere 'al'» |
| **Дублікат дня giornate** | Дозволено, години сумуються |
| **Збереження без клієнта** | Дозволено; `clienteId=null`; «Senza cliente» |
| **`contaMieOreComeCosto=true`, «io» без тарифу** | `costo io=0`, без ефекту; підказка «Imposta una tariffa per te» |
| **`In parte` X > lordo** | Обмежуємо `X ≤ lordo` |
| **Закриття sheet до Salva** | Draft зберігається; повторний `＋` → «Riprendere la bozza?» |

### 6.11 Створення `programmato` — що змінюється

Той самий каркас, інший характер (§6.2). Замість грошей-на-вхід:

```
  Modalità  ( preventivo · ore · giornate · totale )
  ┏━ Quando ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
  ┃  ◷  [ data ]  …або…  [ dal – al ]   ┃   ← пунктирна рамка
  ┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
  + Operai pianificati   (намір участі, без годин)
  ╌╌╌ тут НЕМАЄ грошей-на-вхід ╌╌╌
         ╔══════════════════╗
         ║    PIANIFICA     ║   ← графіт CTA
         ╚══════════════════╝
```

**Що планується (всі числа очікувані):**

| Modalità | Ввід | Показ суми |
|---|---|---|
| `preventivo` | `prezzo` + `periodo` | `previsto € N` (тверда сума) |
| `ore` | очікувані години + тариф-снапшот | `stima ≈ € N` (= ore × tariffa) |
| `giornate` | рядки `data·ore` як план днів | сумарні → `stima ≈ € N` |
| `totale` | очікувана сумарна кількість | `stima ≈ € N` |

> **Канон:** у плані `db.ore` **порожній** — фактичних годин ще нема. Очікувані
> живуть у `durataPrevistaOre` / `oreTotale` / `RigaGiorno` як **намір вводу**.
> Тому `calcoloLavoro.lordo` дало б `0` — суму показує окрема UI-деривація
> `stimaProgrammato(l)` (`src/lib/programmato.ts`, не зберігається): для
> `preventivo` → `{stima:prezzo, esatta:true}` → «`previsto €N`»; для `ore` →
> `{stima:oreAttese×tariffaCli, esatta:false}` → «`stima ≈ €N`». Латунь
> приглушена — повна латунь грошей лише у `svolto`.

**Тверде «ні» у плані:** жодного `incassato`/`da incassare`/`INCASSA`/
`Incassa subito`/`statoIncasso`/штампа `SVOLTO`/✓; **жодного запису в
`db.pagamenti` до конверсії**; `netto` лише як `netto previsto ≈` дрібним у
деталі, ніколи на картці й ніколи без «≈».

### 6.12 Конверсія `programmato → svolto` — перетікання, не новий запис

Конверсія — **той самий `id` дозріває**: пунктир обводиться начисто, синька
теплішає, картка осідає у потік `svolto`. **Не створюється новий `Lavoro`** —
мутується наявний.

**Авто-переноситься (без питань):** `id`, `clienteId`, `titolo`, `descrizione`,
`modo`, `conteggio`, `partecipanti`, `tariffaClienteSnapshot`, `periodo`,
`prezzo`. `fase: da_fare → fatto`.

**Запитується (мінімальний «aркуш дозрівання»):**

| Поле | Дефолт (з плану) |
|---|---|
| `Data` свершення | дата картки / `oggi` |
| Реальні `ore` (modo:ore/giornate/totale) | очікувані — один тап підтвердити |
| `prezzo` (modo:preventivo) | `l.prezzo` з плану |
| Operai: реальні години | з `Operai pianificati` |

> `preventivo`: нуль питань про години — підтвердив дату й суму → готово.

**Створюється при конверсії:** `RegistrazioneOre` на реальні години (тут
очікувані вперше стають фактом); `Pagamento{importoAtteso:lordo,
importoIncassato:0, origine за modo, stato:"in_attesa"}`. Після цього
`calcoloLavoro` працює «по-справжньому» — `incassato` не примусовий `0`,
з'являється кнопка `INCASSA`/`Incassa subito`.

```
КРОК 0 ─ Картка programmato: swipe «✓ segna come svolto» АБО кнопка SVOLTO
            ▼
КРОК 1 ─ Аркуш «matura» морфить вгору з картки; каркас той самий, пунктир
         повзе у суцільне, поля передзаповнені з плану:
            ┌───────────────────────────────────┐
            │  Segna come svolto                 │
            │  Quando?  [ oggi 23 giu ]          │
            │  Ore reali: [ 4 ] ✎  (= очікувані) │
            │  Operai:  io 4h                    │
            │     ╔═════════════════╗            │
            │     ║ CONFERMA SVOLTO ║ ← латунь    │
            │     ╚═════════════════╝            │
            └───────────────────────────────────┘
            ▼
КРОК 2 ─ CONFERMA: атомарна мутація store:
            • fase: da_fare → fatto
            • створити db.ore (реальні години)
            • створити db.pagamenti (atteso=lordo, incassato=0)
            ▼
КРОК 3 ─ АНІМАЦІЯ (той самий id): пунктир→суцільна; синька→тютюн; штамп SVOLTO
         впечатується; кутик «bozza» зникає; картка ОСІДАЄ (тінь→0);
         «stima ≈» → «da incassare €800», з'являється INCASSA
            ▼
КРОК 4 ─ Звичайна svolto-картка тієї ж дати. Наступний крок — INCASSA (опц.).
```

> У `Scheda cliente` рядок **переїжджає** зі смуги `In programma` у `Svolti`
> (shared-element), а не зникає і з'являється. Це дозрівання, не «ще одна
> робота».

**Крайові випадки плану:**

| Випадок | Рішення |
|---|---|
| **Частково зроблено** (`Spezza`) | Аркуш: «Ho fatto solo una parte». Поточний → `svolto` на введену частину; авто-породжується новий `programmato`-`Lavoro` на залишок (питає «коли решта?»). UI-підпис «↳ resto» (доменно — два незалежні `Lavoro`) |
| **Перенесення** (`sposta`) | Тягання картки / `SPOSTA` → змінюється лише `l.data` (+`periodo`); жодних `db.ore`/`pagamenti`; фаза лишається `da_fare` |
| **Скасування** (`annulla`) | tombstone `deleted`; **нуль каскаду** (план не тримав ні годин, ні платежів); «Nessun incasso registrato — si elimina solo la nota» |
| **Прострочений** (`in ritardo`) | UI-дериват: `da_fare && data<oggi`; рамка теракотово-іржава (`critico`), бейдж `da fare · in ritardo · N gg`; спливає вгору на Agenda; лічильник `arretrati` у Dashboard |
| **План у минулому при створенні** | Легітимний → одразу `in ritardo`, не блокуємо |
| **План без клієнта** | Дозволено; тариф питається модалкою лише при конверсії, не при плануванні |

### 6.13 Свідомо за межі v1

- **Редагування `tariffaSnapshot` після створення** — снапшот заморожений;
  перерахунок старих робіт поза v1.
- **`modo:"misto"`** — нові записи не створюють; лише читання legacy-seed.
- **Множинні платежі при створенні** — максимум один `Pagamento`; доплати — з
  екрана роботи / `Soldi`.
- **`acconto_saldo` Preventivo-документ** — окремий флоу; `Crea` фіксує
  факт+гроші, не друкує кошторис.
- **Прив'язка `attrezzi`/`patrimonio`** — поза цим сценарієм.


## 7. «Cliente» — створення, вибір, картка (НАЙГЛИБШЕ)

`Cliente` — коренева сутність домену. До неї ведуть рівно **три двері**, і кожна — повноцінний екран, не «кнопочка збоку»:

- **(A) Новий** (`Crea nuovo`) — `sheet`-форма, мінімум обовʼязкового, прогресивне розкриття.
- **(B) Вибір наявного** (`Scegli esistente`) — окремий екран-список з пошуком/сортуванням; повернення зі **збереженим станом** створення роботи.
- **(C) Картка-«досьє»** (`Scheda`) — деталь: `codice`-ідентичність, calendario, три **розділені** стани оплат, summary.

```
            ┌──────────────────── CLIENTE (radice) ────────────────────┐
   (A) CREA NUOVO          (B) SCEGLI ESISTENTE          (C) SCHEDA (dossier)
   входи:                  входи:                        входи:
   • Anagrafiche → «Nuovo» • Crea → riga Cliente → lista • Anagrafiche → tap riga
   • Crea → riga Cliente   • A зсередини («+ nuovo»)     • Soldi/Dashboard → debitore
     → «+ nuovo» (inline)                                • shared-element з картки lavoro
```

---

### A. СТВОРЕННЯ НОВОГО КЛІЄНТА

#### A.1. Два контексти — один компонент

| Контекст | Звідки | Форма | Після збереження |
|---|---|---|---|
| `inline` (швидкий) | `Crea` → рядок `Cliente` → «`+ Nuovo cliente`» | `sheet` знизу, **1 обовʼязкове поле** (`Nome`); решта згорнута | клієнт **підставляється** в роботу, sheet закривається, **стан `Crea` збережено** |
| `pieno` (повний) | `Anagrafiche` → «`Nuovo`» | той самий `sheet`, секція тарифу розкрита одразу | клієнт у списку `Anagrafiche`; опц. перехід на `Scheda` |

**РІШЕННЯ:** це **той самий компонент-`sheet`** з тим самим набором полів. Різниця лише в контексті входу (передзаповнення, поведінка після save) і скільки секцій розкрито. Жодного дубльованого коду форми.

#### A.2. Поля сутності (з `types.ts` — нічого не вигадано)

| Поле моделі | Ярлик UI | Обовʼязкове? | Канонічне рішення |
|---|---|---|---|
| `nome` | «`Nome`» | **так** (єдине строге) | `trim`, не порожнє |
| `cognome` | «`Cognome`» | ні | `""` для `Azienda` (назву кладе в `nome`) |
| *(тип фіз/юр)* | сегмент «`Persona`»/«`Azienda`» | так (default `Persona`) | **не в моделі** — UI-перемикач ярликів/розкладки; для відображення `cognome===""→azienda`; памʼятається лише в сесії створення |
| `inizialiCodice` | *(не редагується)* | авто | `assegnaIniziali(nome, cognome, db.clienti)` — лічильник колізій `MR→MR1→MR2`; прев'ю штампа |
| `tariffaOraria` | «`Tariffa oraria`» €/h | ні | прогресивно; default `null` (=`0` у розрахунках) |
| `modalitaPredefinita` | «`Come lavori di solito`» | так (default) | `Modalita` default «`A ore`»; вибір `preventivo`/`ore`/`misto` |
| `telefono`/`email`/`luogo` | «`Contatti`» | ні | згорнуто; `luogo` — автокомпліт з наявних |
| `note` | «`Note`» | ні | згорнуто під «`Altro`» |
| `creatoIl`/`id` | — | авто | `inputData(new Date())` / `nuovoId("c")` |

**Поза v1 (явно):** окреме поле `tipo` фіз/юр у моделі; P.IVA/codice fiscale; кілька адрес/контактів; теги дерев. Форма вже згрупована так, щоб докласти без перебудови.

#### A.3. Прогресивне розкриття — «порожнє поле не існує»

```
СЕКЦІЇ ФОРМИ (зверху вниз):
  ┌ ЗАВЖДИ ВИДНО ────────────────────────────────┐
  │ • Сегмент Persona/Azienda                     │
  │ • Nome (+ Cognome для Persona)                │
  │ • Прев'ю штампа codice (live)                 │
  └───────────────────────────────────────────────┘
  ┌ ТРИГЕР-РЯДКИ (згорнуто; тап = розкрити) ──────┐
  │ + Tariffa e modalità   (tariffaOraria+modalità)│
  │ + Contatti             (telefono/email/luogo) │
  │ + Note                 (note)                  │
  └───────────────────────────────────────────────┘
```

- `inline`: усі тригери **згорнуті** → максимальний фокус, шлях у **2 тапи + ім'я**.
- `pieno`: «`Tariffa e modalità`» **розкрита одразу** (в `Anagrafiche` заводиш свідомо), решта згорнуті.
- Розкрита секція не зникає; згорнути назад можна, введене значення зберігається.

**Сегмент `Persona`/`Azienda`** — єдина розвилка-морф (жест segment-slide):

```
[ Persona ]  →  Nome  + Cognome     (дві колонки; inizialiCodice = nome[0]+cognome[0])
[ Azienda ]  →  Ragione sociale     (одне широке поле → nome; cognome="")
```

**РІШЕННЯ (UI-шар, не доменна зміна):** для `Azienda` беремо **перші дві літери `nome`** як ініціали при формуванні аргументів `assegnaIniziali`, щоб штамп був осмислений (`«Verde Vivo» → VV`). Поля морфять height-ом (cross-fade), спільні (tariffa, contatti) лишаються на місці.

#### A.4. `codice parlante` — присвоєння і показ

**РІШЕННЯ:** `codice` рахується, не зберігається; зберігається лише фіксований `inizialiCodice`.

1. Як введено `nome`(+`cognome`) — UI **наживо** рахує `inizialiCodice = assegnaIniziali(...)`.
2. Решта `codice` (`GG-SS-AA`) для нового = **`00-00-00`** (нуль платежів/робіт/років) — чесний стан, не помилка. Повний прев'ю: `MR-00-00-00`.
3. Зберігаємо **тільки `inizialiCodice`**. Повний `codice` дає `codiceCliente(db, id)` при читанні — «оживає» сам із роботами й оплатами.

```
ПРЕВ'Ю ШТАМПА В ФОРМІ (моноширинний, латунь на Carta):
   ╔══════════════════╗
   ║   M R-00-00-00   ║   ← live; II наживо, GG-SS-AA = нулі поки що
   ╚══════════════════╝
   «Il codice cresce da solo: giorni di incasso, spesa media, anni insieme.»
```

#### A.6. Потоки (A)

```

  2. Tap «+ Nuovo» → sheet створення (inline), мінімальний.
```

#### A.7. Крайові випадки (A)

- **Дублікат імені:** **не блокуємо** (два «Mario Rossi» легітимні). Інлайн-натяк під полем «`Esiste già: Mario Rossi · MR-03-12-05`» + мікро-кнопка «`Usa quello`». Ініціали розводяться лічильником (`MR→MR1`) — `codice` унікальний. Інформуємо, не забороняємо.
- **Порожнє/пробіли `nome`:** CTA `disabled`; «`Serve almeno un nome`».
- **`Azienda` без `cognome`:** норма; `cognome=""`.
- **Тариф `0`/порожній:** дозволено (`tariffaOraria=null`); валідно для роботи `a preventivo`.
- **Скасування (`✕`/свайп-вниз):** якщо введено щось — «`Annullare? I dati non saranno salvati`». Порожня форма закривається без питання.
- **Inline зі `Crea`:** гарантія — **стан роботи ніколи не скидається** при відкритті/закритті sheet.

#### B.2. Пошук і сортування — канон

- **Пошук** фільтрує по `nome`, `cognome`, `inizialiCodice` (шукати «MR»). Регістронезалежно, `includes`. Порожній запит — повний список.
- **Сортування:** «`Recenti`» (default у B — за `creatoIl` спадно, щойно створений нагорі) · «`A-Z`» (за `nome`, it-локаль) · «`Saldo`» (за `riepilogoCliente().saldoDaIncassare` спадно — те саме, що default `Anagrafiche`, консистентність).
- Метадані підрядка: `tariffa/h · numeroLavori lavori`; `saldoDaIncassare>0` → латунний бейдж суми; `numeroLavori===0` → бейдж «`nuovo`».

#### B.3. Потік (B) і повернення стану

```
1. Crea, рядок Cliente → tap → екран B.
2. (опц.) пошук / зміна сортування.
3a. Tap на рядок → клієнт ПІДСТАВЛЕНИЙ у Crea → B закривається →
    назад у Crea з УСІМ збереженим станом (fase, modo, conteggio, operai,
    ore-рядки, spese, periodo…).
3b. Або «+ Nuovo cliente» → A-inline → після save повертає В ТОЙ САМИЙ Crea
    з новим клієнтом підставленим (B пропускається).
4. Якщо в Crea вже був клієнт — у B він ✓; tap по іншому замінює.
```

**КАНОН-ГАРАНТІЯ:** стан `Crea` живе вище за стек цих екранів і **не демонтується**.

#### B.4. Крайові випадки (B)

- **Жодного клієнта:** велика картка «`+ Nuovo cliente`» + `EmptyState` «`Ancora nessun cliente. Crea il primo.`».
- **Пошук без результатів:** «`Nessun cliente per "xyz"`» + кнопка «`+ Nuovo cliente`», що **передзаповнює `nome`** запитом.
- **Один клієнт:** список нормально (не автообираємо — користувач має бачити вибір).
- **Back без вибору:** в `Crea`, рядок `Cliente` лишається попереднім; стан не чіпається.

### C. SCHEDA CLIENTE (картка-«досьє»)

#### C.2. Calendario — дні робіт (зроблені і планові)

Не повний місячний грід (це дім `Agenda`), а **компактна часова стрічка**: минуле тепле-важке-суцільне, майбутнє прохолодне-легке-пунктирне.

```
┌─ Lavori nel tempo ───────────────────────────────┐
│  ‹ 2025 ›                                         │
│  GIU   ● ● ◌            (● fatto · ◌ programmato) │

│  prossimo: ◌ Lun 28 — Potatura olivi (da fare)    │ ← найближчий da_fare виділено
└───────────────────────────────────────────────────┘
```

- Джерело: `db.lavori[clienteId]` за `data` (місяць/день). Тон дня з `fase`/`stato`.
- Найближча планова (`da_fare`, мін. майбутня `data`) — окремим рядком «що далі».
- Tap по дню → провал у деталь роботи (shared-element `codice` назад).
- **v1-спрощення:** горизонт ~останні 12 міс + усі майбутні; повний календар — `Agenda`.

```
│ (3) «DA FARE / PROGRAMMATO» — креслення/чернетка, прохолодний    │
│     ┌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌┐             │

```

**Канон віднесення (все деривується, без дублювання):**

| Потік | Джерело | Правило (домен) |
|---|---|---|
| (1) `pagato` | `db.pagamenti[clienteId]` | `statoCalcolato(p) === "pagato"` |
| (2) `da incassare` | `db.pagamenti[clienteId]` | `importoAtteso − importoIncassato > 0.005` → `in_attesa`/`in_ritardo`; `in_ritardo` показує `giorniRitardo(dataScadenza)` |
| (3) `da fare` | `db.lavori[clienteId]` | `fase === "da_fare"` (fallback `stato !== "fatto"`); ще без `pagamento` |

- Сегмент-фільтр «`Tutto`/`Da incassare`/`Da fare`» (жест segment-slide): «`Tutto`» — три потоки секціями; інші — лише свій.
- Частковий (`parziale`, `0 < importoIncassato < importoAtteso`) живе в (2) з підписом «`parziale: incassato X / atteso Y`» + бейдж `info`.

#### C.4. ЗАКРИТТЯ БУДЬ-ЯКОЇ ОПЛАТИ — повна функціональність (інлайн)

Будь-який рядок (2) несе повноцінну інкасацію **інлайн** (бо сценарій простий). Узгоджено з «`Incassa subito`» і `Soldi`.

```
Tap «Incassa ▾» → інлайн-розкриття:
┌─ Incassa: Saldo lavoro siepe ──────────────────────┐
│  da incassare:  450 €                               │
│  ┌─────────────────────────────────────────────┐   │
│  │   ✓  Incassa tutto · 450 €     (Ottone)     │   │ ← 1) ПОВНА (головна дія)
│  └─────────────────────────────────────────────┘   │
│  Importo parziale:  [   …   ] €   [ Conferma ]      │ ← 2) ЧАСТКОВА
│  rapidi:  [ 100 ] [ 200 ] [ metà 225 ]              │ ← 3) ШВИДКІ
│  Data incasso: [ oggi 24/06 ▾ ]   Metodo: [▾]       │ ← default oggi; метод
└─────────────────────────────────────────────────────┘
```

**Семантика (мапиться на домен один-в-один):**

- **Повна** («`Incassa tutto`»): `importoIncassato=importoAtteso`, `dataIncasso=oggi`, `stato="pagato"`. Рядок **морфить** із (2) у (1) зі штампом `SALDATO` (хаптик; гроші не підстрибують). Зʼявляється в `movimenti` (incasso +) і `riepilogoMese.incassato`.
- **Часткова**: `importoIncassato += введене` (clamp ≤ `importoAtteso`), `dataIncasso=oggi`. Стан → `parziale` (лишається в (2)). Якщо доводить до повної — як повна.
- **Швидкі**: `[100][200][metà]` підставляють у поле часткової; «`metà`» = `arrotonda(residuo/2)`.
- **Дата/метод**: `dataIncasso` (default `oggi`), `metodo` (`contanti`/`bonifico`/`altro`).
- **Каскад-канон:** інкасація має сенс на `fase==="fatto"`. Якщо рядок (3) `da_fare` хочуть «`Incassa subito`» — спершу пропонуємо позначити `fatto` (морф (3)→(1)), потім інкасуємо.

**РІШЕННЯ:** усі дії йдуть через дію стора (репозиторій) — UI ніколи не пише в базу напряму. Тут зафіксована лише семантика.

#### C.5. RIEPILOGO по клієнту (мінімалістичний)

З `riepilogoCliente(db, clienteId)` — нічого не вигадано, нічого зайвого на поверхні:

```
┌─ Riepilogo (гравірований, 4 числа) ─────────────┐
│  da incassare      incassato      lavori   ore  │
│   saldoDa-          totale-       numero-  ore-  │
│   Incassare         Incassato     Lavori   Totali│

   margine · valoreFatturabile · costoManodopera · spese
```

- **Поверхня (завжди):** `saldoDaIncassare` (латунь якщо >0), `totaleIncassato`, `numeroLavori`, `oreTotali`.
- **Глибина (за тапом «`Dettagli economici`»):** `margine`, `valoreFatturabile`, `costoManodopera`, `spese` — економіка майстра за прогресивним розкриттям. Жодних дубльованих агрегатів.

#### C.6. Шапка-ідентичність і shared-element `codice`

- `codice = codiceCliente(db, id)` — **геройський елемент** картки. Під ним розшифровка `leggiCodice(codice)`: «`paga in ~{giorni} gg · ~{spesaMedia} €/lavoro · {anni} anni`».
- **Shared-element:** з будь-якої картки `lavoro` (де `codice` клієнта — підпис) `codice` **летить** у цей штамп (continuity), і назад.
- Меню «`⋯`»: «`Modifica`» (→ A-sheet у режимі edit, той самий компонент), «`Archivia`», «`Elimina`».

#### C.7. Потоки (C)

```
ВХІД: Anagrafiche → tap рядок («відкриття папки») · Soldi/Dashboard → debitore ·
      shared-element: картка lavoro → tap codice/ім'я → Scheda
ДІЇ:  «+ Nuovo lavoro per {nome}» (sticky) → Crea з передзаповненим Cliente
      рядок da incassare → «Incassa» інлайн (C.4)
      рядок da fare → tap → деталь роботи; або «Pianifica»
      рядок pagato → tap → деталь (читання історії)
      ✏ → редагувати (A-sheet, edit-режим)
```

#### C.8. Крайові випадки (C)

- **Клієнт без жодної роботи** (`numeroLavori===0`): картка не порожня — штамп `XX-00-00-00`, `riepilogo` нулями, calendario `EmptyState` «`Ancora nessun lavoro`», великий заклик «`+ Crea il primo lavoro per {nome}`». Три потоки — свої `EmptyState`. Валідний стан.
- **Клієнт лиш із планом** (тільки (3)): (1)/(2) порожні з мʼякими `EmptyState`; calendario лише `◌`.
- **Боржник у простроченні** (`in_ritardo`): рядок (2) `critico` (teracotta, не неон), бейдж «`IN RITARDO {giorniRitardo} gg`», піднятий нагору (2).
- **`Elimina` зі звʼязками** (роботи/оплати/ore): **блокуємо**. Модалка «`Questo cliente ha {n} lavori e {m} pagamenti. Non puoi eliminarlo: puoi archiviarlo.`» → пропонуємо `Archivia`.
- **`Archivia` (РІШЕННЯ v1):** у моделі немає `deleted`/`attivo` — архів = **фільтр-конвенція** до появи tombstone-флага (`updatedAt`+`deleted` закладені для майбутнього). Архівований не зникає з домену: `codice`/історія лишаються, він поза головним списком `Anagrafiche`. Розархівувати — тривіально.
- **`Elimina` без жодного звʼязку:** дозволено жорстко, з підтвердженням «`Eliminare {nome}? Azione irreversibile`».
- **Дублікати в шапці:** `codice` (`MR` vs `MR1`) розрізняє однозначно — призначення лічильника ініціалів.

## 8. «Operaio» — створення, картка, дії

`operaio` (canone-сутність `Operatore`) — це **той, хто працював руками**. За
замовчуванням це **я сам**: власник, `ruolo: "titolare"`, у UI відомий як
`«io»`. Поряд інколи працює найманий `collaboratore`. Уся squadra-механіка
(`libroOperatore`, `compensi`, `da pagare`) уже живе в домені — цей розділ дає до
неї **фасад**: як `operaio` створюється, обирається й живе на власній картці.

Дві наскрізні істини, з яких усе виводиться:

- **`operatoreIo`** присутній завжди (`ruolo === "titolare"`, fallback — перший
  оператор). Це `«io»`.
- **`contaMieOreComeCosto`** (default `false`): мої години входять у `lordo`
  (виручку), але **не** в `costoCollaboratori`. Тобто **мій час за замовчуванням =
  чистий прибуток**, а не витрата (детально — §8.6).

### 8.1 Поля сутності (форма створення/редагування)

Мапиться 1:1 на `Operatore`. Форма свідомо коротка — `operaio` створюється «на
льоту» з контексту роботи.

| Поле UI (it) | Canone-поле | Тип / правило | Видимість |
|---|---|---|---|
| `Nome` | `nome` | рядок, обовʼязковий | завжди |
| `Ruolo` | `ruolo` | `«Titolare»` / `«Collaboratore»` | завжди; для `«io»` заблоковано на `titolare` |
| `Costo orario` | `tariffaOraria` | €/год, **це `costo` (моя витрата)**, опц. | завжди, з підказкою |
| `Telefono` | `telefono` | рядок, опц. | тригер-рядок «＋ aggiungi telefono» |
| `Note` | `note` | текст, опц. | тригер-рядок |
| `Attivo` | `attivo` | boolean, default `true` | лише в редагуванні |
| (службове) | `id`, `creatoIl` | `nuovoId("op")`, ISO-дата | автоматично |

**Рішення — мікрокопі тарифу.** Підпис під `Costo orario`:
«Quanto riconosci a questo operaio per ogni ora. È un **costo** per te, non un
prezzo per il cliente.» Це розрізняє `tariffaOraria` оператора (costo) від
`tariffaOraria` клієнта (ricavo) — у домені це різні поля на різних сутностях,
але назва однакова, тож UI мусить це проговорити.

**Рішення — тариф для `«io»`.** За seed `titolare.tariffaOraria = null`. Форма
`«io»` показує `Costo orario` **порожнім** з підписом: «Di solito il tuo tempo
non è un costo — lo lasci vuoto.» Заповнення поля **не** вмикає
`contaMieOreComeCosto` — це окремий перемикач на рівні роботи (§8.6); тариф
`«io»` лише заготовлене значення на випадок його ввімкнення.

### 8.2 Створення / вибір — два входи + лист «Scegli operaio»

```
ВХІД 1 — з КОНТЕКСТУ РОБОТИ:
  Crea registrazione → рядок «Operaio: «io» ▾» → тап → лист «Scegli operaio»

ВХІД 2 — з ANAGRAFICHE:
  Anagrafiche → tab [ Operai ] → «＋ nuovo operaio» → той самий лист
```

Обидва входи відкривають **той самий** bottom-sheet; різниться лише
продовження (з роботи — повертає у форму з доданим учасником; з Anagrafiche —
відкриває картку нового `operaio`).

```
┌─────────────────────────────────────────────┐
│  Scegli operaio                          ✕   │
│  ─────────────────────────────────────────   │
│  🔎 cerca…                                    │
│                                              │
│  ● «io»  (titolare)            ✓ già nel      │  ← завжди першим, доданий за дефолтом
│                                  lavoro       │
│  ─────────────────────────────────────────   │
│  ○ Luca      collaboratore · 16 €/h          │  ← активні (attivo=true), сорт. за nome
│  ○ Marco G.  collaboratore · 18 €/h          │
│  ○ Anna      collaboratore · — €/h           │  ← без тарифу → costo 0, попередження
│                                              │
│  ⋯ mostra non attivi (2)                      │  ← attivo=false сховані за тригером
│  ─────────────────────────────────────────   │
│  ［ ＋ Crea nuovo operaio ］                   │  ← інлайн-створення (нижче)
└─────────────────────────────────────────────┘
```

- `«io»` завжди першим, помічений `✓ già nel lavoro` (канон: `partecipanti`
  мінімум = `«io»`).
- Тап по рядку → **toggle** участі. Тап «Crea nuovo» → інлайн-форма (зростає
  знизу, default `Ruolo = Collaboratore`); `Salva` → `nuovoId("op")`,
  `attivo:true`, повертає в лист із новим `operaio` **вже вибраним**; форма
  роботи не скидається.

**Снапшот тарифу — КАНОН.** При додаванні в роботу в `partecipanti` пишеться
`tariffaSnapshot = operatore.tariffaOraria` на момент створення. Пізніша зміна
`Costo orario` оператора **не** перераховує минулі роботи (`partecipantiCalc`
бере `p.tariffaSnapshot ?? tariffaOf(id)`).

**Рішення — попередження про порожній тариф.** Якщо додають `operaio` із
`tariffaOraria == null` у роботу `ore`/`totale`/`per_giorni`, при `Salva` мʼяка
модалка: «Anna non ha un costo orario. Le sue ore varranno **0 €** come costo.»
→ `[ Imposta costo ]` / `[ Va bene così ]`.

### 8.3 Картка `operaio` («scheda operaio»)

Характер — **«особисте досьє» людини команди**. Акцент **не** на `codice
parlante` (у операя його немає), а на **діях** і балансі `da pagare`.
Статистика навмисно МІНІМАЛЬНА (маленька стрічка зверху), головний фокус — ДІЇ.

```
┌───────────────────────────────────────────────────────┐
│ ◀                                            ⋯ modifica │  ← назад / меню (edit, attiva/disattiva)
│   ╭─ вушко-папка ─╮                                     │
│   │  L            │                                     │  ← ініціал-жетон (латунь)
│   ╰───────────────╯                                     │
│   Luca                                  ● collaboratore │  ← імʼя (серіф) + badge ruolo
│   16 €/h · costo                                        │  ← costo orario, моноширинний
│                                                         │
│ ┌─── statistiche (piccolo, НЕ на весь екран) ─────────┐ │
│ │  ⌚ 42 h     💶 saldato 100 €     ⬤ da pagare 572 € │ │  ← ТРИ числа одним рядком-стрічкою
│ └─────────────────────────────────────────────────────┘ │
│                                                         │
│ ════════════ AZIONI (фокус екрана) ═════════════════════│
│  ┌─────────────────────────┐ ┌───────────────────────┐ │
│  │  ＋ Aggiungi lavoro      │ │  💶 Paga operaio       │ │  ← дві головні дії
│  │     svolto · oggi        │ │     salda «da pagare»  │ │
│  └─────────────────────────┘ └───────────────────────┘ │
│  ［ ◷ Pianifica lavoro futuro (programmato) ］          │  ← окрема дія: майбутнє
│                                                         │
│ ════════════ STORICO (per calendario) ══════════════════│
│  giugno 2026                                            │  ← групування по місяцях (як Agenda)
│  ┌─ 18 mar ─────────────────────────────────────────┐  │
│  │ Rossi · potatura      4 h    svolto    64 €      │  │  ← рядок-квитанція (теплий, суцільний)
│  └──────────────────────────────────────────────────┘  │
│  ┌╴ 28 ven ╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶┐  │
│  ╎ Verdi · impianto     —      programmato         ╎  │  ← пунктир = майбутнє (легкий, прохолодний)
│  └╴╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶╶┘  │
└───────────────────────────────────────────────────────┘
```

**Стрічка статистики — джерела з `libroOperatore(db, operatoreId)`** (нічого не
дублюється):

| Число | Джерело | Формула |
|---|---|---|
| `⌚ N h` | `libro.ore` | `arrotonda(Σ годин з db.ore)` |
| `💶 saldato X €` | `libro.pagato` | `arrotonda(Σ compensi)` |
| `⬤ da pagare Y €` | `libro.saldo` | `arrotonda(dovuto − pagato)`, `dovuto = ore × tariffaOraria` |

Badge на `da pagare` — `libro.stato`: `«saldato»` / `«parziale»` / `«da pagare»`
(тон з `STATO_COMPENSO_TONO`).

**Рішення для картки `«io»`.** За замовчуванням `da pagare` **не** показується як
борг (мій час не витрата) — стрічка стає:
```
┌─ statistiche («io») ────────────────────────────────┐
│  ⌚ 128 h     🌿 mio tempo · non è un costo           │  ← замість «da pagare» — пояснення
└──────────────────────────────────────────────────────┘
```
Якщо по якихось роботах увімкнено `contaMieOreComeCosto`, стрічка `«io»`
показує `da pagare` **лише по тих роботах** (рідкісний кейс — §8.6).

**`storico` per calendario:** групування по місяцях (`meseAnnoIT`), сорт. за
спаданням дати; рядок — робота, де `operaio` був учасником
(`partecipanti.collaboratoreId`, або legacy — `db.ore` за `operatoreId`); поля:
`data` · `cliente · titolo` · його `ore` · badge `fase` · його `costo`
(`ore × tariffaSnapshot`). Фактура: `svolto` — суцільна тепла рамка, моноширинні
числа; `programmato` — пунктир, `«—»` замість годин. Тап → провал у детальний
екран роботи (картка не редагує роботу, лише точка входу).

### 8.4 Дві дії-створення з протилежних кінців `fase`

| | `svolto` (минуле) | `programmato` (майбутнє) |
|---|---|---|
| `fase` | `"fatto"` | `"da_fare"` |
| Створює `RegistrazioneOre`? | **так** (реальні години) | **ні** (лише `durataPrevistaOre`) |
| Впливає на `da pagare` одразу? | **так** (росте `dovuto`) | **ні** (доки не svolto) |
| Дата за замовчуванням | `oggi` | майбутній день |
| Характер UI | теплий, важкий, штамп, суцільна рамка | прохолодний, легкий, олівець, пунктир |
| Час | смуга необовʼязкова | `oraInizio/oraFine` доречні (слот в Agenda) |

**`Aggiungi lavoro` (svolto):** форма з передзаповненим `operaio`, штамп `SVOLTO`;
`Cliente` (тригер-рядок), `Data` = `oggi`, `Titolo`, `Modalità`
(`preventivo·ore·giornate·totale`), `Ore di Luca`, live-`costo`. `Salva` пише
`fase:"fatto"`, `partecipanti` (`«io»` + Luca зі `tariffaSnapshot`) і
**`RegistrazioneOre`** `(lavoroId, operatoreId, ore, data)` — джерело фактичних
годин; одразу росте `da pagare`.

**`Pianifica lavoro futuro` (programmato):** пунктирна форма; `Data` =
майбутній день, опц. `Ora 09:00–13:00`, `Durata prevista` (стима). Підказка:
«Le ore reali si registrano a lavoro fatto.» `Pianifica` пише `fase:"da_fare"`,
`durataPrevistaOre`, `partecipanti` — **але НЕ `RegistrazioneOre`**; `libro.ore`
не росте, `da pagare` не змінюється; у `storico` рядок пунктиром, `«—»` годин.

**Місток `programmato → svolto`:** свайп-дія `segna come svolto` на пунктирному
рядку → морф пунктир→суцільне → `fase:"fatto"` → **мінізапит** «Quante ore ha
fatto Luca?» (бо саме тут створюється `RegistrazioneOre`) → `da pagare` росте.
План не коштує нічого, доки не «відштампований» зробленим.

### 8.5 Третя дія — `Paga operaio`

Інлайн-дія (не окремий екран), узгоджена з Soldi-зрізом
(`da incassare ↔ da pagare`).

```
┌─── Paga operaio · Luca ──────────────────────┐  ← характер «банка/sejf»: латунь, темно
│  Da pagare:        572 €                      │  ← libro.saldo
│  Importo:  [ 572 ]    ［ tutto ］              │  ← default = весь saldo
│  Data:     [ 24/06/2026 ]                     │
│  Metodo:   ( contanti · bonifico · altro )    │  ← MetodoPagamento
│  Periodo:  [ 2026-06 ]  (opц.)                │
│            ［ ▒ Paga ▒ ］                       │  ← вагома «печатка», хаптик-штамп
└───────────────────────────────────────────────┘
```

Пише `CompensoOperatore` (`{ operatoreId, importo, data, periodo?, metodo? }`).
Наслідки: `libro.pagato ↑` → `libro.saldo ↓` → badge `«parziale»`/`«saldato»`;
зʼявляється в `movimenti` як `compenso` (`segno:-1`, «Compenso squadra»);
зменшує `riepilogoMese.daPagareSquadra`. Гроші **НА ВИХІД** — протилежний
напрям до `incasso` клієнта.

### 8.6 `«io»`-власник проти найманого `operaio` — серце канону

Різниця тримається на **одному прапорці на рівні роботи** —
`contaMieOreComeCosto` — і правилі виключення в `calcoloLavoro`:
```
escludi = (p.collaboratoreId === ioId) && !l.contaMieOreComeCosto
costo   = escludi ? 0 : arrotonda(p.ore × p.tariffa)
```
`lordo` рахує **всі** години однаково (мій час **завжди** заробляє).
`costoCollaboratori` рахує тільки невиключених; `«io»` за замовчуванням
виключений → мій `costo = 0`. `netto = lordo − spese − costoCollaboratori`.

| | `«io»` (titolare, default) | найманий `operaio` |
|---|---|---|
| Години в `lordo` (виручці) | **входять** | **входять** |
| Години в `costoCollaboratori` | **0** (не витрата) | `ore × tariffaSnapshot` |
| Зменшують `netto`? | **ні** | **так** |
| `da pagare` на картці | **немає** (я собі не плачу) | **є** (`dovuto − pagato`) |
| Сенс | мій час = чистий прибуток | його час = виплата, гроші НА ВИХІД |

**Приклад.** Робота `ore`, клієнт 28 €/год, 10 год я + 5 год Luca (16 €/год):
`lordo = 15 × 28 = 420 €`; `costoCollaboratori = Luca 80 € + я 0 = 80 €`;
`netto = 420 − 80 = 340 €`. На картці Luca `da pagare ↑ 80 €`; на моїй —
нічого.

**Перемикач для рідкісного випадку** (порахувати «справжню» собівартість):
```
Crea/dettaglio lavoro → секція учасників:
  ☐ Conta le mie ore come costo
     ⓘ Di norma il tuo tempo è guadagno, non spesa.
```
При `true`: мій `costo = ore × tariffaSnapshot` (тариф `«io»` з §8.1), `netto ↓`,
на картці `«io»` зʼявляється `da pagare` **по цих роботах**. Усвідомлений,
рідкісний вибір — захований за тригером.

### 8.7 Потоки (flows)

```
FLOW 1 · operaio з роботи: Crea registrazione → «Operaio ▾» → лист →
  «＋ Crea nuovo» → Salva → доданий зі tariffaSnapshot → Salva lavoro →
  RegistrazioneOre → da pagare ↑.
FLOW 2 · operaio з Anagrafiche: [Operai] → ＋ nuovo → Salva → scheda (storico
  порожній, da pagare 0).
FLOW 3 · картка svolto: ＋ Aggiungi lavoro → cliente+oggi+ore → Salva →
  рядок svolto → da pagare ↑.
FLOW 4 · картка programmato: ◷ Pianifica → cliente+futuro+stima → Pianifica →
  пунктир → da pagare без змін.
FLOW 5 · місток: storico → swipe «segna come svolto» → «quante ore?» →
  RegistrazioneOre → da pagare ↑.
FLOW 6 · виплата: 💶 Paga operaio → importo(=saldo)+metodo → CompensoOperatore →
  saldato/parziale → movimenti(compenso).
FLOW 7 · io→costo: dettaglio lavoro → ☑ Conta le mie ore come costo →
  netto ↓ → scheda «io» показує da pagare по цій роботі.
```

### 8.8 Крайові випадки

| # | Випадок | Рішення |
|---|---|---|
| E1 | `operaio` без `tariffaOraria` у роботі `ore` | `costo = 0`; при `Salva` модалка «imposta costo?». `da pagare = 0` доки не задано тариф. |
| E2 | Тариф змінили **після** створення роботи | Минулі не міняються (`tariffaSnapshot` заморожений); нові беруть новий тариф. |
| E3 | `«io»` пробують видалити / `attivo:false` | Заборонено. `operatoreIo` має існувати завжди. `disattiva` для `titolare` прихована; спроба → toast «Il titolare non si può disattivare». |
| E4 | `programmato` з годинами-слотом, без `ore` | Нормально: `durataPrevistaOre` — стима, `db.ore` порожні; у `libro` не рахується, доки не `svolto`. |
| E5 | Виплата > `da pagare` (переплата) | Дозволити + попередити «+X €»; `saldo` відʼємний → badge `«saldato»`, у статистиці «credito X €». |
| E6 | Legacy-робота без `partecipanti` | `partecipantiCalc` legacy-гілка: з `db.ore` за `operatoreId` + `l.operatoreId`, тариф актуальний (без снапшоту). |
| E7 | `attivo:false` зі старими роботами | Лишається в `storico` та `libro`; зникає лише з листа вибору нової роботи (за тригером). |
| E8 | Картка `«io»` з `contaMieOreComeCosto` на частині робіт | `da pagare` лише по роботах із прапорцем; підпис «da pagare (solo lavori a costo pieno)». |
| E9 | Видалення `operaio` з відкритим `da pagare` | Блокувати hard-delete; пропонувати `disattiva`. Мʼяке видалення — для синку, баланс лишається в історії. |
| E10 | Дублікати імен (`Marco G.`/`Marco R.`) | `nome` не унікальний; розрізнення в листі — `ruolo · costo`. Codice для операя не вводимо. |

## 9. «Soldi» — центр грошей

`Soldi` 💶 — окремий блок нижнього таб-бара (`Agenda · ＋ · Soldi · Anagrafiche`).
Це не «вид даних», а **дія**: інкасую / плачу. `Dashboard` свідомо **немає** в
таб-барі — вхід у нього лише з підсумкового блоку `Soldi` (тап) або свайпом
ліворуч від `Agenda`.

Характер екрана: `Soldi` = «сейф / гросбух» — найтемніша панель, `carta-incasso`
(темно-зелено-латунна «шкіра книги»), числа табличні «гравіровані», підтвердження =
вагома «печатка» + хаптик-`stampa`. Гроші не підстрибують.

Уся арифметика бере значення **тільки** з доменного шару (`movimenti.ts`,
`conti.ts`, `lavoro-calc.ts`, `squadra.ts`) — `Soldi` нічого не рахує сам.

### 9.1. Загальна структура

```
┌──────────────────────────────────────────────────────────┐
│  ░░ carta-incasso — «шкіра гросбуха», латунний кант ░░     │
│   SOLDI                                  «giugno 2026» ▾   │ ← селектор періоду
│  ──────────────────────────────────────────────────────── │
│  ╔══════════ BLOCCO RIEPILOGO (3 dashboard) ═══════════╗   │ §9.3
│  ║  guadagnato      incassato      da incassare         ║   │ тап → Dashboard
│  ╚══════════════════════════════════════════════════════╝  │
│   [ da incassare ]──────slide──────[ da pagare ]           │ §9.2 сегмент
│  ──────────────────────────────────────────────────────── │
│   ▼ список рядків активного режиму (реєстр-ліновка)        │
└──────────────────────────────────────────────────────────┘
```

Сегмент `da incassare ⇄ da pagare` — той самий вивчений жест, що
`svolto/programmato` у `Agenda`: спільне лишається, відмінне морфить.

### 9.2. Два режими — `da incassare` ⇄ `da pagare`

Дві сторони одного гросбуха. Каркас списку (геометрія рядка, відступи, лінійки)
**спільний**; різниться напрям грошей, субʼєкти, палітра, дія. Перехід — морфінг:
`list` не перебудовується, лише «перефарбовується», стрілка потоку розвертається.

**`da incassare` — «клієнти винні мені».** Напрям ▼ всередину сейфа. Субʼєкти —
`clienti`. Джерело: `debitori(db)` (з `conti.ts`) — клієнти з відкритим залишком,
сортовані за спаданням `saldo`; під кожним — його відкриті `pagamenti`
(`importoAtteso − importoIncassato > 0.005`). Палітра: акцент `positivo`
(оливково-латунний), стрілка `▼ entra`. Прострочення
(`statoCalcolato(p)==="in_ritardo"`) — тон `critico` (теракота), бейдж
`in ritardo · NN gg`.

```
┌──────────────────────────────────────────────────────────┐
│  [ DA INCASSARE ]·············  da pagare                  │
│  ──────────────────────────────────────────────────────── │
│   totale da incassare              ▼  € 1 116,00           │ ← Σ debitori.saldo
│  ════════════════════════════════════════════════════════ │   (вся база)
│   MR1 · Maria Ricci                          ▼ € 266,00    │ ← debitore
│   ┌── codice MR1-..-..-.. ───────────────────────────┐    │
│   │  «Potatura» · a ore · 9,5 h × 28 €     € 266,00   │    │ рядок = pagamento
│   │  emesso 12/06 · scad 30/06        [ Incassa ▸ ]   │    │ → потік E1
│   └────────────────────────────────────────────────────┘  │
│   GB · Giulia Bianchi          ⚠ in ritardo · 8 gg        │
│   ┌────────────────────────────────────────────────────┐  │
│   │  «Siepe» · saldo          € 450,00   teracotta      │  │
│   │  emesso 04/06 · scad 15/06        [ Incassa ▸ ]     │  │
│   └────────────────────────────────────────────────────┘  │
│  ··········································  perforazione ·· │
│   [ + Incasso Subito ]   ← §9.4: новий чистий preventivo   │
└──────────────────────────────────────────────────────────┘
```

Кліки: тап по `codice`/імені → `ClienteScheda` (`codice` летить
shared-element-ом); тап по тілу → `Cantiere`; `Incassa` → інлайн-швидкодія (E1).

**`da pagare` — «виплатити операям і собі».** Напрям ▲ назовні. Субʼєкти —
`operatori`. Джерело: для кожного `o` → `libroOperatore(db, o.id)` (з `squadra.ts`),
показуємо де `saldo > 0.005`. `daPagareSquadra = Σ max(0, libroOperatore(o).saldo)`.
Палітра: акцент `attenzione` (вохра), стрілка `▲ esce`, **та сама геометрія** —
це й створює відчуття «перетікання».

Рішення щодо «собі»: `operatoreIo` (titolare) має `tariffaOraria: null` (`=0`),
тож `dovuto=0`, `saldo=0` — за замовчуванням «io» **не** зʼявляється у `da pagare`
(мої години = прибуток, не борг). Але блок «собі» існує окремим рядком завжди —
як **prelievo** (вилучення власника з каси), не як `compenso` за годинами (E3).

```
┌──────────────────────────────────────────────────────────┐
│   da incassare  ·············  [ DA PAGARE ]               │
│  ──────────────────────────────────────────────────────── │
│   totale da pagare squadra         ▲  € 52,00             │ ← daPagareSquadra
│  ════════════════════════════════════════════════════════ │
│   ▸ SQUADRA (operai)                                       │
│   Luca · collaboratore            parziale  ▲ € 52,00     │ ← libroOperatore.saldo
│   ┌────────────────────────────────────────────────────┐  │   (152 − 100)
│   │  9,5 h × 16 €  =  dovuto € 152,00                   │  │
│   │  già pagato € 100,00                                │  │
│   │  ▸ per cliente: Ricci 9,5 h · € 152                 │  │ ← perCliente
│   │                              [  Paga operaio ▸ ]    │  │ → потік E2
│   └────────────────────────────────────────────────────┘  │
│   ▸ IO (titolare) — prelievo                               │
│   ┌────────────────────────────────────────────────────┐  │
│   │  cassa disponibile  € 998,00   (incassato − uscite) │  │ ← saldo del mese
│   │  «non è un costo, è un prelievo»  [  Preleva ▸ ]    │  │ → потік E3
│   └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**Перехід (морфінг каркаса):** рядки зберігають висоту/відступи (не перебудова,
лише «перефарбовування»); стрілка потоку розвертається `▼ → ▲`; субʼєкти
cross-fade `clienti → operai`; палітра `positivo → attenzione`; заголовок-сума
morph `«da incassare» ↔ «da pagare squadra»`.

### 9.3. Підсумковий блок — три пропорційні дашборди

Три рівноправні, стилістично пропорційні плитки (однакова сітка, латунний кант,
табличні числа). Тап по **будь-якій** → екран `Dashboard` (його немає в таб-барі) —
один з двох входів (другий — свайп ліворуч від `Agenda`). Канонічне джерело —
`riepilogoMese(db, anno, mese)` для обраного періоду.

| Плитка | Ярлик (it) | Значення | Формула (домен) |
|---|---|---|---|
| 1 | `guadagnato` | зароблено за місяць | `Σ calcoloLavoro(l).lordo` по `lavori` з `fase==="fatto"` і `data` у місяці |
| 2 | `incassato` | отримано | `riepilogoMese.incassato` (`Σ importoIncassato` де `dataIncasso` у місяці) |
| 3 | `da incassare` | залишок до отримання | `riepilogoMese.daIncassare` (`Σ max(0, atteso − incassato)`, **по всій базі**) |

Канонічне розрізнення: плитка 1 `guadagnato` = **нарахована виручка** (lordo за
зробленими роботами місяця), плитка 2 `incassato` = **реально отримані гроші**
місяця, плитка 3 = **відкритий залишок усієї бази**. Тому 1 і 2 — різні числа
(заробив у червні, інкасую в липні), і це навмисно видно. Плитка 3 — по всій базі
(борг не «належить місяцю»).

```
╔════════════ BLOCCO RIEPILOGO ·· tap → Dashboard ════════════╗
║ ┌───────────────┐ ┌───────────────┐ ┌───────────────┐       ║
║ │  guadagnato   │ │   incassato   │ │  da incassare │       ║
║ │     mese      │ │     mese      │ │   (residuo)   │       ║
║ │   € 1 250     │ │    € 720      │ │   € 1 116     │       ║
║ │  ▔▔▔ latteo  │ │  ▔▔▔ latlat  │ │  ▔▔▔ teracotta│       ║
║ └───────────────┘ └───────────────┘ └───────────────┘       ║
║   нарахов. lordo    реально отримано   борг (вся база)      ║
╚═════════════════════════════════════════════════════════════╝
        тап будь-куди ▸ зум-морф у Dashboard (shared element)
```

Акцент-тон лише на третій плитці (`attenzione`/`critico` якщо є прострочені), щоб
око одразу бачило «що недоотримано». Перетікання у Dashboard — `shared element`:
блок розгортається в повноекранний Dashboard, числа «летять» на місце.

### 9.4. `Incasso Subito` — вся оплата одразу, без вписування годин

**Сценарій:** чистий `preventivo` (фіксована ціна, години не рахуються), робота
зроблена, клієнт платить **повну суму одразу**. Один жест закриває все — ні годин,
ні поетапного `pagamento`.

Де кнопка:
1. У `Soldi`, режим `da incassare`, низ списку: `[ + Incasso Subito ]` (нова робота з нуля).
2. На картці роботи в `Agenda` та в `Cantiere`: при `modo==="preventivo"` без відкритого `pagamento` — інлайн `Incassa subito`.
3. У листі `Crea registrazione` (FAB ＋): при `svolto + preventivo` — чекбокс `☑ incassa subito` (коміт `e845784`).

```
[ + Incasso Subito ]  (sheet знизу, морф з кнопки)
┌──────────────────────────────────────────────┐
│  INCASSO SUBITO            «a preventivo»      │
│  ──────────────────────────────────────────── │
│  Cliente:  ( + scegli / nuovo )   ← тригер-рядок│
│  Descrizione: «Potatura giardino»             │
│  Importo:   € [  800,00  ]        ← prezzo     │
│  Data:      [ oggi 24/06 ]                     │
│  Metodo:    ( contanti · bonifico · altro )    │
│  ┌──────────────────────────────────────────┐ │
│  │  Si crea:                                 │ │
│  │  • lavoro «fatto · preventivo» € 800      │ │
│  │  • pagamento incassato € 800 (oggi)       │ │
│  │  niente ore da inserire.                  │ │
│  └──────────────────────────────────────────┘ │
│                  [  Incassa € 800  ▸ ]         │ ← STAMPA «incassato»
└──────────────────────────────────────────────┘
```

**Що створюється** (атомарно, одна транзакція store):

1. `Lavoro`: `fase:"fatto"`, `modo:"preventivo"`, `conteggio:"totale"`, `prezzo:800`,
   `partecipanti:[{ collaboratoreId: ioId, tariffaSnapshot: 0 }]` (мінімум «io», без
   `oreTotale`), `contaMieOreComeCosto:false`, `data:oggi`, legacy-синхро
   `stato:"fatto"`/`tipoCompenso:"preventivo"`. **`db.ore` НЕ створюється** — у цьому сенс.
2. `Pagamento` (одразу зінкасований): `lavoroId`, `clienteId`, `origine:"preventivo"`,
   `importoAtteso:800`, `importoIncassato:800`, `dataEmissione:oggi`, `dataIncasso:oggi`,
   `stato:"pagato"`. Метод зберігається в `note` (домен не має поля `metodo`).

Каскад (усе виводиться): `calcoloLavoro` → `lordo=800`, `incassato=800`,
`statoIncasso="pagato"`, `oreTotali=0`, `netto=800`; `statoCalcolato → "pagato"`;
зʼявляється в `movimenti` як `incasso +800`, у `riepilogoMese.incassato`, у плитці
`incassato`; **не** зʼявляється в `da incassare` (`daIncassare=0`).

Крайовий «клієнт новий»: тригер-рядок `Cliente` потребує `nome/cognome` →
`assegnaIniziali` дає `inizialiCodice`; модалка `imposta tariffa` **не** потрібна
(preventivo не рахує години). Без жодного pagato `codice = XX-00-00-00` — це нормально.

### 9.5. Провал у статистику історичних записів

Будь-який рядок у `Soldi`/Dashboard/картці клієнта → провал у `Records`
(`Records.tsx`) — повна історія записів. Це **не агрегати**, а перелік фактів-подій
із фільтрами. Кожен запис показує канонічно: коли зроблено/створено, години, який
preventivo, дати, оплачено vs ні, io vs інший operaio.

```
┌──────────────────────────────────────────────────────────┐
│  STORICO                    [ filtri ▾ ] [ cliente ▾ ]    │
│  ──────────────────────────────────────────────────────── │
│  filtri: periodo · cliente · operaio (io/altro) ·          │
│          modo (preventivo/ore) · stato incasso · fase      │
│  ──────────────────────────────────────────────────────── │
│  12/06/2026 · «Potatura» · MR1 Maria Ricci                │
│  ┌────────────────────────────────────────────────────┐  │
│  │ fatto · a ore · 9,5 h        creato 12/06          │  │ ← creatoIl vs data
│  │ partecipanti:  io 0 h · Luca 9,5 h × 16 €          │  │ ← io vs altro (chip)
│  │ lordo € 266 · costo € 152 · netto € 114            │  │ ← calcoloLavoro
│  │ incassato € 0 / 266        ◷ non pagato (parziale) │  │ ← statoIncasso (тон)
│  └────────────────────────────────────────────────────┘  │
│  10/06/2026 · «Siepe» · MR Mario Rossi                    │
│  ┌────────────────────────────────────────────────────┐  │
│  │ fatto · a preventivo  € 800       creato 09/06     │  │ ← який preventivo
│  │ partecipanti: io 6 h (profitto)                    │  │ ← мої години = прибуток
│  │ ✓ incassato € 800 / 800     pagato 10/06           │  │ ← оплачено + дата
│  │   preventivo: «Cifra unica» €800                   │  │ ← TipoPreventivo
│  └────────────────────────────────────────────────────┘  │
│  ▸ tap su riga → Cantiere (dettaglio lavoro)              │
└──────────────────────────────────────────────────────────┘
```

Звідки беруться поля: **коли зроблено** = `l.data`, **коли створено** = `l.creatoIl`
(показуємо обидва); **години** = `calcoloLavoro.oreTotali` + розбивка
`partecipanti[].ore`; **який preventivo** = `db.preventivi[lavoroId].tipo`
(`unico`/`acconto_saldo`) + суми, для `modo==="ore"` — позначка «a ore»; **дати** =
`data`, `creatoIl`, `pagamento.dataEmissione`, `dataScadenza`, `dataIncasso`;
**оплачено** = `calcoloLavoro.statoIncasso` (`non_pagato`/`parziale`/`pagato`);
**io vs operaio** = `partecipanti[]`: чіп `io` (з «profitto», якщо `costo===0`) проти
чіпів операїв (`ore × tariffa = costo`). Сортування за спаданням `data`; усе
виводиться при читанні.

### 9.6. Потоки оплат

Усі мутації — через дії store (`store.ts`), щоб під бекенд міняти лише реалізацію.
Підтвердження = «печатка» (`carta-incasso`), хаптик-`stampa`, без відскоку.

**E1 — Інкасувати від клієнта (повна / часткова / швидка).** Кнопка `Incassa` на
рядку `da incassare`. Ціль — `pagamentoApertoLavoro(db, lavoroId)` (перший платіж з
`residuo > 0.005`).

```
[ Incassa ▸ ]  (інлайн-розгортання в рядку)
┌────────────────────────────────────────────────────┐
│  Incassa da  Maria Ricci · «Potatura»               │
│  atteso € 266,00 · già incassato € 0,00             │
│  residuo  € 266,00                                  │
│  importo:  € [ 266,00 ]   [ tutto ] [ ½ ] [ altro ] │ ← швидкі суми
│  data: [ oggi 24/06 ]   metodo: (contanti·bonifico) │
│           [  Conferma incasso € 266  ▸ ]            │ ← STAMPA
└────────────────────────────────────────────────────┘
   ↓ store: incassaPagamento(pagId, importo, data)
   importoIncassato += importo; se piena → dataIncasso=data, stato="pagato"
```

Стани (з `statoCalcolato` + `statoIncasso`):
- `importo === residuo` → **повна**: `importoIncassato=importoAtteso`,
  `dataIncasso=data`, `stato="pagato"` → рядок зникає, бейдж `✓ pagato`, `festa` (UI).
- `0 < importo < residuo` → **часткова**: `stato` лишається `in_attesa`/`in_ritardo`,
  `statoIncasso="parziale"`, рядок лишається з оновленим залишком.
- **швидка** (`[tutto]`) = повна в один тап без редагування суми.

**E2 — Виплатити operaio.** Кнопка `Paga operaio`. Джерело боргу —
`libroOperatore(db, operatoreId).saldo`.

```
[ Paga operaio ▸ ]
┌────────────────────────────────────────────────────┐
│  Paga  Luca · collaboratore                         │
│  dovuto € 152,00 · pagato € 100,00 · saldo € 52,00  │
│  importo: € [ 52,00 ]   [ saldo ] [ altro ]         │
│  data: [ oggi ]  periodo [ 2026-06 ]  metodo: (…)   │
│          [  Conferma compenso € 52  ▸ ]            │ ← STAMPA
└────────────────────────────────────────────────────┘
   ↓ store: pagaOperatore({operatoreId, importo, data, periodo, metodo})
   crea CompensoOperatore → libroOperatore ricalcola saldo/stato
```

Стани (з `statoCompenso(dovuto, pagato)`): `pagato >= dovuto` → **`saldato`** →
рядок зникає; `0 < pagato < dovuto` → **`parziale`** → лишається. Зʼявляється в
`movimenti` як `compenso −importo`, у `riepilogoMese.compensi`/`uscite`.

**E3 — Виплатити собі (`prelievo`).** `io`/titolare. Це **не** `CompensoOperatore`
за годинами (бо `tariffaOraria(io)=null`, мої години = прибуток), а **вилучення з
каси**.

```
[ Preleva ▸ ]
┌────────────────────────────────────────────────────┐
│  Preleva (io · titolare)                            │
│  cassa del mese  € 998,00   (incassato − uscite)    │
│  importo: € [ 500,00 ]    [ tutto ]    data:[ oggi ]│
│  «non è un costo del lavoro, è un prelievo»         │
│          [  Conferma prelievo € 500  ▸ ]           │
└────────────────────────────────────────────────────┘
   ↓ store: prelievoTitolare({importo, data})
```

v1-рішення: `Database` не має сутності «prelievo», тож у v1 реалізуємо як
`CompensoOperatore` з `operatoreId = io` + `note: "prelievo"`. Коректно входить у
`movimenti`/`riepilogoMese.uscite` як вихід коштів і не ламає `libroOperatore` (бо
`dovuto(io)=0`, `saldo` стає негативним — «взято наперед», у `da pagare` не
показуємо, бо фільтр `saldo > 0`). **Поза v1:** окрема сутність `Prelievo` з власним
реєстром.

### 9.7. Крайові випадки

**Переплата (`importoIncassato > importoAtteso`).** `statoCalcolato → "pagato"`;
`calcoloLavoro.daIncassare = max(0, lordo − incassato) = 0`.

```
┌────────────────────────────────────────────────────┐
│  ⚠ Importo supera il residuo                        │
│  residuo € 266 · stai incassando € 300              │
│  eccedenza € 34,00                                  │
│  ( ) registra solo € 266 (consigliato)              │
│  (•) registra € 300 — eccedenza come acconto futuro │ ← canon v1: дозволити
│           [  Conferma  ▸ ]                          │
└────────────────────────────────────────────────────┘
```
v1-рішення: дозволяємо переплату (округлення/чайові), показуємо `eccedenza`.
Надлишок лишається в `importoIncassato` як кредит; на наступному `pagamento`
пропонуємо «usa credito € 34».

**Часткова (`parziale`).** Рядок лишається в `da incassare`/`da pagare` з оновленим
залишком, бейдж `parziale`; `daIncassare`/`daPagareSquadra` зменшуються рівно на
внесену суму. Жодного «зникнення», поки `residuo > 0.005`.

**Повернення / сторно (`reso` / `storno`).**

```
┌────────────────────────────────────────────────────┐
│  STORNA incasso                                     │
│  pagamento € 266 incassato 23/06                    │
│  importo da stornare: € [ 266,00 ]   [ tutto ]      │
│  «verrà sottratto dall'incassato»                   │
│           [  Conferma storno  ▸ ]                   │ ← richiede tap lungo
└────────────────────────────────────────────────────┘
   ↓ store: stornaIncasso(pagId, importo)
   importoIncassato −= importo; se 0 → dataIncasso=null, stato="in_attesa"
```
v1-рішення: `storno` = не видалення запису, а зменшення `importoIncassato`
(`Math.max(0, …)`); якщо `0` → `dataIncasso=null`, `stato` повертається до
`in_attesa`/`in_ritardo` (за `dataScadenza`). У `movimenti` не додаємо окремого
негативного рядка у v1. Вимагає підтвердження довгим натиском (деструктивна грошова
дія). **Поза v1:** окремий `movimento` типу `storno` для повного аудит-сліду.

**Інші зафіксовані рішення:**
- `da incassare` для роботи `fase==="da_fare"`: `incassato=0`, але `Incassa` спершу
  пропонує `segna come svolto` (морф пунктир → суцільне), потім інкасує.
- Клієнт без тарифу + `modo "ore"`: `tariffaCli=0` → `lordo=0` → нічого інкасувати;
  при `Incassa` спливає модалка `imposta tariffa`.
- Період без даних: плитки (9.3) показують `€ 0,00`, списки — порожній стан
  «nessun movimento questo mese» (паперова заглушка, не помилка).

## 10. «Що буде, якщо…», поза межами v1, глосарій UA↔IT

> Останній розділ — це **запобіжники й словник**. Тут зведено всі крайові випадки з
> канонічними рішеннями (не питаннями), явно окреслено межу v1 і дано повний словник
> UA↔IT, щоб жоден термін у решті документа не лишався двозначним.

---

### 10.1. Каталог крайових випадків — рішення, не питання

Кожен рядок — це факт, який може статись у полі, і **єдине канонічне рішення**. Якщо
екран чи дія не покривають випадок — правий цей каталог.

| № | Випадок | Канонічне РІШЕННЯ |
|---|---|---|
| E1 | **Нуль годин** у режимі `ore`/`giornate`/`totale` | Зберегти **не можна**: CTA `Salva` неактивна, поки `Σ ore = 0`. Підпис «`inserisci almeno un'ora`». `giornate` із порожнім останнім рядком ігнорує його при сумі. |
| E2 | **Робота без тарифу** (`ore`, клієнт без `tariffaOraria`) | При `Salva` спливає модалка `Imposta tariffa` («`Quanto vale un'ora di questo lavoro?`») з опцією `Salva senza tariffa` → фіксуємо лише `ore`, `lordo = 0`, робота стає видимою як «`da tariffare`». Тариф можна задати пізніше — економіка перерахується. У `preventivo` тариф не потрібен ніколи. |
| E3 | **Операй без тарифу** (`tariffaSnapshot` відсутній) | Дозволено: `costo = 0`, маркер `⚠` у списку вибору. Робота зберігається, `da pagare` цього операя на ній = `0`. Це свідома гілка «допоміг безкоштовно / тариф уточню потім», не помилка. |
| E4 | **Переплата** (`Incassa` суму > `daIncassare`) | Поле інкасо **не пускає** ввести більше за залишок: верхня межа = `daIncassare`, швидкі чіпи `[metà]`/суми клампляться. Окремої сутності «кредит клієнта» у v1 немає — переплата неможлива за конструкцією. |
| E5 | **Часткова оплата** | Канонічна, не крайова: `Pagamento` на частину суми, робота лишається в потоці `da incassare` із залишком, бейдж `parziale`. Сума залишку = `lordo − Σ pagamenti(lavoro)`. Деривується, не зберігається. |
| E6 | **Прострочений план** (`programmato`, дата в минулому, ще `da_fare`) | Картка лишається `programmato`, але отримує маркер `◷ scaduto` (теракота/іржа, «земляний» сигнал). Домен **не** переводить її у `svolto` автоматично — лише майстер дозріває її дією `Segna come svolto`. Гроші не чіпаються (план завжди `incassato = 0`). |
| E7 | **Дублікат клієнта** (нове ім'я збігається з наявним) | Збереження **дозволено** (тезки реальні), але форма показує м'яке попередження «`esiste già: MR-03-12-05 · Mario Rossi`» з кнопкою `Usa quello`. Колізія `codice` вирішується лічильником: другий `MR` → `MR1`, третій → `MR2`. Жодного жорсткого блокування. |
| E8 | **Видалення клієнта зі зв'язками** | **Заборонено.** Тільки `Archivia` (tombstone, ховається зі списків, лишається джерелом істини для історії й тоталів). Видалити фізично можна лише клієнта **без** робіт/оплат. |
| E9 | **Видалення операя зі зв'язками** | Аналогічно E8: `Archivia` замість видалення. `«io»` (`titolare`) **неможливо** ні видалити, ні архівувати. |
| E10 | **Видалення `«io»`** | Неможливо: `titolare` — системний корінь економіки (мій час = прибуток). Не показується в дій-меню видалення взагалі. |
| E11 | **Часткова конверсія `programmato → svolto`** (план із кількох `giornate`, виконано не всі дні) | Дозрівання — **атомарне на всю роботу**: `Segna come svolto` морфить увесь запис. Якщо реально зроблено лише частину — майстер редагує `giornate` (прибирає невиконані дні) **перед** позначкою, або лишає план і створює окремий `svolto`. Часткового дозрівання однієї роботи v1 не має. |
| E12 | **Зміна тарифу клієнта/операя після створення роботи** | Старі роботи **не перераховуються**: на кожній заморожено `tariffaSnapshot` у момент створення. Нова ставка діє лише на нові роботи. Це канон собівартості (Документ 1, § Operatore). |
| E13 | **Робота без клієнта** (швидкий запис) | Дозволено: `clienteId = null`, робота видима як «`senza cliente`». Економіка рахується нормально (тариф питається через E2). Клієнта можна прив'язати пізніше редагуванням. |
| E14 | **Інкасо плану** (`programmato`) | Неможливо: домен примусово `incassato = 0` для `fase: "da_fare"`. Кнопки `Incassa` на плановій картці немає — спершу дозрівання у `svolto`. |
| E15 | **`Incasso Subito` без суми** | CTA неактивна, поки `importo = 0`. Атомарність: `Lavoro` + `Pagamento` народжуються разом або не народжуються зовсім. |
| E16 | **Виплата собі** (`«io»` у `da pagare`) | Не «зарплата», а `prelievo` — вилучення з каси. `«io»` ніколи не висить боржником (мій час = прибуток); сума доступна до вилучення = накопичена `cassa`, не «нарахований борг». |
| E17 | **Офлайн / мертва зона звʼязку** | Усе працює локально (local-first, IndexedDB). Запис зберігається миттєво на пристрої; синк — відкладений і непомітний. Жодна дія не блокується відсутністю мережі. |
| E18 | **Конфлікт синку** (той самий запис змінено у двох місцях) | Поза v1 (синку ще немає), але **шов закладено**: `UUID` + `updatedAt` + tombstone. Майбутня стратегія — last-write-wins за `updatedAt`. |

```
   ПРИНЦИП КРАЙОВИХ ВИПАДКІВ
   ┌─────────────────────────────────────────────────────────┐
   │  факт первинний → ніколи не втрачаємо подію              │
   │  число похідне  → переплата/розсинх неможливі за конст. │
   │  замість DELETE → ARCHIVIA (tombstone)                   │
   │  замість блоку  → м'яке попередження + дія-обхід         │
   │  «io» недоторканний · план не приймає грошей             │
   └─────────────────────────────────────────────────────────┘
```

---

### 10.2. Явно **поза межами v1**

Усе нижче — свідомо відкладене. Це не «забули», а «не зараз». Архітектура лишає шов,
щоб додати без переписування екранів.

- **Реальний бекенд і синк IndexedDB ↔ Postgres.** v1 — чистий local-first; шов
  (`UUID`/`updatedAt`/tombstone, шар-репозиторій) закладено, увімкнення — поза v1.
- **Мультикористувач / кілька пристроїв з онлайн-синком.** Один майстер, один пристрій.
- **Розвʼязання конфліктів синку** (E18) — лише після появи самого синку.
- **Кредит/аванс клієнта** (переплата як сутність, E4) — немає балансу «клієнт винен мінус».
- **Часткове дозрівання однієї роботи** (E11) — дозрівання завжди атомарне.
- **Звіти/експорт у PDF, фіскальні документи, fattura elettronica** — це зошит, не бухгалтерія.
- **Податки, ПДВ (`IVA`), нарахування внесків** — економіка ремесла, не податковий облік.
- **Мультивалюта** — лише `€`.
- **Багатомовний UI** — інтерфейс лише італійською (це ідентичність, не локалізація).
- **Повторювані/циклічні роботи** (`ricorrenza`) — кожен `programmato` створюється вручну.
- **Сповіщення/нагадування push** — `Agenda` показує, але не «дзвонить».
- **Ролі/права доступу** понад `titolare`/`collaboratore` — немає менеджерів, адмінів.
- **Вкладення файлів/фото до роботи** — лише текст і числа у v1.
- **Аналітика-графіки на `Dashboard`** понад зрізи `Clienti`/`Operai` — без діаграм трендів.
- **Інтеграції** (календарі, банк, месенджери) — повна автономність зошита.

```
  v1 МАЄ                          ПОЗА v1
  ┌──────────────────────┐        ┌──────────────────────────┐
  │ local-first зошит     │        │ бекенд · синк · конфлікти │
  │ факти → похідні числа │   ⟶   │ кредит клієнта · IVA      │
  │ 4 modalità · 3 потоки │        │ PDF/fattura · мультивал.  │
  │ Incasso Subito        │        │ push · ролі · вкладення   │
  │ archivia (no delete)  │        │ ricorrenza · інтеграції   │
  └──────────────────────┘        └──────────────────────────┘
            шов закладено ──────────────▶ вмикається без переписування
```

### 10.3. Повний глосарій UA↔IT

Усі канонічні терміни. Колонка **IT** — те, що буквально проростає в екрани (`моноширинне`
= рядок коду/сутність/енумерація; «в лапках» = видимий ярлик UI).

#### Сутності й корені домену

| UA (зміст) | IT (канон) | Примітка |
|---|---|---|
| клієнт | `Cliente` / «cliente» | корінь історії; має `codice parlante` |
| операй / руки | `Operatore` / «operaio» | корінь типу й таблиць; ярлик у UI — «operaio» |
| я / власник | `«io»` · `titolare` | мій час = прибуток, не витрата; недоторканний |
| напарник | `collaboratore` | найманий; його час = реальні гроші на вихід |
| робота / зроблене | `Lavoro` | центральна сутність розрахунків |
| оплата | `Pagamento` | факт грошей **до мене** |
| компенсація операю | `CompensoOperatore` | факт грошей **від мене** команді |
| витрата | `spesa` / `spese` | напр. бензин; зменшує `netto` |
| привіт. список годин | `libroOperatore` | джерело `ore`/`saldato`/`da pagare` операя |

#### Розвилка часу та фази

| UA | IT | Примітка |
|---|---|---|
| зроблене / минуле | `svolto` (`fase: "fatto"`) | продуктовий ярлик `svolto`; значення поля `fase:"fatto"`; штамп `SVOLTO` |
| заплановане / майбутнє | `programmato` (`fase: "da_fare"`) | ярлик `programmato`; значення поля `fase:"da_fare"`; `bozza`, `incassato = 0` |
| фільтр «майбутнє» в UI | «Da fare» (ярлик) ↔ `da_fare` (токен) | видимий ярлик — Title Case з пробілом; технічний токен — моноширинний з підкресленням |
| дозрівання плану | «Segna come svolto» | той самий запис морфить, не новий |
| прострочений план | `scaduto` | маркер `◷`, дата минула, ще `da_fare` (E6) |

#### Чотири `modalità di calcolo`

| UA | IT | Сенс |
|---|---|---|
| домовлена сума | `preventivo` | «домовились на суму», години неважливі |
| години сьогодні | `ore` | один день, одне число × тариф |
| кілька днів | `giornate` | рядки `{дата · ore}`, що сумуються |
| загалом за період | `totale` | сума годин + `periodo`, дні неважливі |

#### Економіка

| UA | IT | Формула / сенс |
|---|---|---|
| виручка | `lordo` | `preventivo`-сума **або** `ore × tariffa` |
| собівартість команди | `costoCollaboratori` | `Σ (ore_i × tariffaSnapshot_i)`; мій costo = 0 |
| чистий | `netto` | `lordo − spese − costoCollaboratori` |
| тариф клієнта | `tariffaOraria` | `€/h`, заморожується снапшотом у роботу |
| заморожений тариф | `tariffaSnapshot` | копія тарифу в момент створення (E12) |
| перемикач «мій час як витрата» | «conta le mie ore come costo» | рідкісний; за замовч. вимкнено |

#### Гроші: напрями й стани

| UA | IT | Напрям |
|---|---|---|
| отримано | `incassato` | реально зінкасовано |
| до отримання | `da incassare` / `saldoDaIncassare` | клієнти винні мені (**до мене**) |
| нараховано | `guadagnato` | виручка нарахована (не конче отримана) |
| до виплати | `da pagare` | я винен команді (**від мене**) |
| повна оплата | «Incassa tutto» · «saldato» | робота повністю закрита |
| часткова | `parziale` | залишок `lordo − Σ pagamenti` (E5) |
| прострочено | «in ritardo» | теракота/іржа, «земляний» сигнал |
| вилучення з каси | `prelievo` | виплата собі — не зарплата (E16) |
| швидкий цикл | «Incasso Subito» | атомарно `Lavoro` + `Pagamento` повна сума |

#### Ідентичність і код

| UA | IT | Сенс |
|---|---|---|
| промовистий код | `codice parlante` (`II-GG-SS-AA`) | інвентарний жетон клієнта, росте з історії |
| ініціали | `II` | напр. `MR`; колізія → `MR1`, `MR2` |
| дні до оплати | `GG` | середня прострочка |
| середні витрати/роботу | `SS` | у сотнях `€` |
| років разом | `AA` | стаж стосунків |
| новий клієнт | `MR-00-00-00` · «nuovo» | «обіцянка», не порожнеча |

#### Екрани / навігація

| UA | IT | Роль |
|---|---|---|
| дім-календар | `Agenda` | центр світу, градієнт часу, `targhetta` |
| створення запису | «Crea registrazione» / `Crea` | єдина точка народження фактів |
| досьє клієнта | «Scheda cliente» | `codice` + 3 потоки |
| досьє операя | «Scheda operaio» | без `codice`, фокус на діях |
| центр грошей | «Soldi» | сейф; `da incassare ⇄ da pagare` |
| зріз згори | «Dashboard» | `Clienti`/`Operai`, поза таб-баром |
| довідники | «Anagrafiche» | `Clienti` · `Operai` |
| вибір клієнта | «Scegli cliente» | push-екран зі збереженням стану `Crea` |
| вибір операя | «Scegli operaio» | той самий патерн |

#### Дрібні маркери, дії, обʼєкти

| UA | IT | Примітка |
|---|---|---|
| бирка дати | `targhetta` | латунний жетон-герой, sticky в `Agenda` |
| штамп «зроблено» | `SVOLTO` / «SALDATO» | відбиток печатки на минулому |
| чернетка | `bozza` | кутик на плановій картці |
| архівувати | «Archivia» | замість видалення (E8/E9) |
| учасники роботи | `partecipanti` | масив `{operatore, ore, tariffaSnapshot}` |
| мітка періоду | `periodo` | «за який відрізок» у `preventivo`/`totale` |
| задати тариф | «Imposta tariffa» | модалка при `ore` без тарифу (E2) |
| без тарифу | «Salva senza tariffa» | фіксуємо години, гроші пізніше |
| персона / фірма | `Persona` / `Azienda` | розвилка форми клієнта |
| назва фірми | `Ragione sociale` | замість `Nome + Cognome` |
